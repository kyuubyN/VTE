# VTE — Vector Tensor Engine

VTE is an inference engine for running LLMs on AMD GPUs on Windows, built directly on top of the HIP runtime instead of any existing inference framework. There's no llama.cpp, no ONNX Runtime, no vLLM, no PyTorch underneath it — VTE parses the GGUF file itself, generates the HIP C++ kernels itself, compiles them with `hipcc` at runtime, and drives `amdhip64.dll` through a hand-written ctypes bridge. It currently runs two architectures — **Qwen2.5-1.5B-Instruct** (Q4_K_M/Q6_K) and **IBM Granite 4.1 3B** (Q8_0) — selectable at runtime (CLI or the desktop UI's model dropdown), and the whole project has been developed and measured on a single consumer card: an RX 7600 (RDNA3, gfx1102, 8GB VRAM).

The reason to build this from scratch rather than wrapping an existing runtime was to have full control over every byte moved between VRAM and the ALUs on a GPU that has neither the memory bandwidth nor the CU count of a datacenter part — and to make every optimization decision based on an actual measurement on this specific hardware, not on what works on an MI300X or an RTX 4090. That discipline ("measure, don't guess") shows up throughout the codebase and is documented below alongside the bugs it caught.

Granite was added specifically as a real, measured data point for a question the project had only been guessing at before: does VTE's architecture scale past a 1.5B toy model, or does the hand-rolled kernel/memory-management approach fall over past a certain size? Granite 3B in Q8_0 is roughly 2x the parameters of Qwen and uses a genuinely different architecture (different RoPE convention, four scale factors Qwen doesn't have, different quantization format) — not just a bigger version of the same thing. It runs correctly, fits comfortably in 8GB VRAM (~4.8GB including margin), and reaches ~89% of llama.cpp/Ollama's throughput on the same hardware for *both* models (see "Performance" and "Multi-architecture support" below) — real evidence, not a hypothesis, that the architecture generalizes.

As of this writing, the engine does single-sequence decode at a **stable ~100 tok/s from the first token to the last** on Qwen2.5-1.5B (250-token generation, first-25 and last-25 averages within 1.5% of each other) and batched decode (multiple sequences in lockstep) peaking at ~200 tok/s aggregate throughput at batch size 4. The single-sequence number climbed from an initial ~41 tok/s baseline across several rounds of profiling in this repository's history — not primarily by rewriting GEMV kernels, but by finding and removing non-GPU costs and non-algorithmic GPU costs one measurement at a time: a CPU-side sampler and an eager (non-graph-captured) LM Head launch that together cost more than the GPU's own 28-layer forward pass; a `flash_attention_kernel` using only 12 of the RX 7600's 32 CUs, whose serial per-position cost made long generations degrade from ~70 tok/s down toward ~50 tok/s the longer they ran; and, once that fix made the decode tick fast enough, a fixed keep-alive pulse (itself added to solve an earlier, real problem) that had become the largest remaining artificial cost in the loop. See "Bugs found during development" below for the full story of each, including a real Windows TDR crash that turned out to have nothing to do with the GPU. (Granite 4.1 3B, the second and much larger model, has its own performance and bug history — see "Multi-architecture support: adding Granite 4.1 3B" further down — including the same `~89%` gap-to-llama.cpp being consistent across both models despite their architectural differences.)

Batched decode was independently re-measured and reconfirmed after a full repository reorganization (200.7 tok/s aggregate / 50.2 tok/s per sequence at batch_size=4, over 100 decode ticks) — these numbers reflect the current state of the code, not a one-off historical run. That said, the ~200 tok/s figure is a GPU-only measurement (HIP Graph replay in isolation); the same eager-LM-Head/sampler overhead found in single-sequence decode was also present in batched decode, and applying the same fix there is documented in "Bugs found during development" below, alongside a correction to how large a win it actually was.

Worth stating plainly, because it's the whole point of measuring against Ollama at all (see "VTE vs. llama.cpp/Ollama" below): the code doing the dispatching here is **Python**, not C++. Ollama's actual inference math runs in llama.cpp — hand-tuned C++ GEMV kernels with years of upstream optimization behind them, called through a thin Go server. VTE's Python layer builds the compute graph, resolves kernel arguments, and drives every HIP launch through ctypes — the same language usually blamed for making GPU dispatch too slow to compete. Landing at **~89% of llama.cpp's throughput, on both architectures independently**, with that much interpreter overhead in the loop, is the actual evidence for this project's core bet: that CPU-side dispatch overhead is an engineering problem you can profile and remove (HIP Graphs, sampler restricted to top-k, LM Head captured into the graph — see "Bugs found during development"), not an inherent tax Python has to pay. The remaining ~11% gap is real GEMV-kernel-efficiency headroom, not "Python being Python."

## Benchmark: VTE vs. Ollama (llama.cpp)

Final comparison result, same GGUF files on disk for both engines, same prompt, `temperature=0` (full methodology under "Performance" below):

| Model | VTE | Ollama (llama.cpp) | VTE / Ollama |
|---|---|---|---|
| Qwen2.5 1.5B (Q4_K_M) | 101.68 tok/s (9.84 ms/tok) | 114.67 tok/s (8.72 ms/tok) | 88.7% |
| Granite 4.1 3B (Q8_0) | 45.66 tok/s (21.90 ms/tok) | 51.46 tok/s (19.43 ms/tok) | 88.7% |

Notably consistent — VTE delivers ~89% of llama.cpp's throughput on both models, an identical ratio despite very different architectures. That's a solid result for an engine built from scratch, in Python, against a mature engine with years of C++ optimization behind it (see the "Python vs. C++ dispatch" note further down for why that comparison matters).

## Quick start

The fastest way to talk to a model is the desktop UI (chat + live GPU telemetry). This is the condensed version — see "Getting started" at the end of this document for the plain Python API and more detail on each step.

**Dependencies:**
- Windows 10/11, an AMD RDNA2/RDNA3 GPU (RX 6000/7000 series), Python 3.10+
- AMD's HIP SDK (ROCm 6.4) — installs `amdhip64.dll` and `hipcc`
- Microsoft C/C++ Build Tools (`cl.exe`) — `hipcc` needs it to find MSVC/Windows SDK headers on Windows (VTE injects the right `PATH`/`INCLUDE` automatically; you just need the Build Tools installed)
- A `.gguf` model file — Qwen2.5-1.5B-Instruct (`Q4_K_M`/`Q6_K`) or IBM Granite 4.1 3B (`Q8_0`) — placed in a `Model/` folder at the project root

**Install and run:**
```bash
python -m venv .venv
.venv\Scripts\activate          # Windows
pip install -e .[dev]
vte-ui
```
`pip install -e .[dev]` registers `vte` as an editable package (so `import vte` and the `vte-ui` command work from anywhere afterward), and `vte-ui` launches the Flet desktop app. Pick the model from the dropdown in the app itself — switching between Qwen and Granite at runtime, mid-session, is supported (see "Multi-architecture support" below for how that's kept from breaking an in-flight chat).

If `vte-ui` isn't on `PATH` for some reason, `python -m vte.ui.app` is the equivalent (note the `-m` — running `python vte/ui/app.py` directly fails with `ModuleNotFoundError`, since that invocation never imports `vte` as a package).

## Performance

| Stage | Throughput (batch=1) | What changed |
|---|---|---|
| Naive per-thread GEMV | ~18.8 tok/s | Baseline: one thread per output element, no coalescing |
| Coalesced GEMV | ~31.5 tok/s | Threads in a block cooperate on one output neuron, coalesced weight reads |
| In-kernel Q4_K/Q6_K dequantization | ~37.9 tok/s | FFN weights dequantized inline during the GEMV instead of a separate dequant pass ("No-Sync Direct Unpack") |
| QKV Two-Pass Split-K fusion | ~41.0 tok/s | Fills all 32 CUs instead of leaving most idle — see "QKV fusion" below |
| Vectorized sampler | ~62–70 tok/s | Top-k/top-p/softmax/argsort restricted to the ~50 surviving candidates instead of the full 151936-token vocabulary — see bug entry below |
| LM Head captured into the HIP Graph | ~71.7 tok/s | Eliminated the last eager (non-graph) kernel launch in the decode loop — see bug entry below |
| Split-KV (Flash-Decoding) attention | ~78–79 tok/s, and *flat* across generation length | Fills far more of the 32 CUs during attention instead of 12; fixes a decline that used to reach ~50.8 tok/s by token 250 — see bug entry below |
| Keep-alive pulse removed (`VTE_KEEPALIVE_PULSE_MS=0.0`, default) | **~100 tok/s** | A 2ms/tick safety pulse (added to stop WDDM's DPM from downclocking between bursts) became the largest artificial cost once the tick above got fast enough that DPM never gets an idle window long enough to act — see bug entry below |

None of the last four rows touched a GEMV kernel's core math — each came from profiling the actual per-token wall-clock budget (not just "the GPU part") and finding real, measurable non-algorithmic costs: CPU-side dispatch overhead, GPU occupancy left on the table, and finally a fixed-cost safety mechanism whose own justification had quietly expired. The GPU-only decode graph itself measures ~130 tok/s in isolation (via HIP Graph replay timing) — the ~100 tok/s end-to-end figure is now close enough to that ceiling that further gains need actual kernel-level work (GEMV efficiency, WMMA), not more overhead-hunting.

Batched decode (`generate_batch`, same-length prompts, lockstep) — GPU-only, HIP Graph replay in isolation (no sampler, no logits readback):

| Batch size | Aggregate tok/s | Efficiency vs. ideal linear scaling |
|---|---|---|
| 1 | 110.7 | — |
| 2 | 165.3 | 75% |
| 4 | 204.2 (peak) | 46% |
| 8 | 193.5 (regresses) | 22% |

The batch=8 regression is attributed to Infinity Cache (32MB) thrashing: FFN weights are ~27.5MB per layer (~88% of per-layer traffic), and at batch=8 the working set for weight reuse across the batch no longer fits comfortably alongside everything else competing for the cache. This hasn't been fixed yet — see the open items at the end of this document.

**Real end-to-end `generate_batch()` throughput at batch_size=4** (sampler + logits readback included, the number that actually matters for a caller): **127.6 tok/s aggregate** (31.9 tok/s/sequence), up from 118.9 tok/s after capturing the LM Head into the batched HIP Graph (same technique as single-sequence decode, adapted to `[batch_size, vocab_size]` geometry) — a real but modest 7.4% gain, smaller than a naive Amdahl's-law projection suggested. See "Bugs found during development" for why: most of the eagerly-measured LM Head cost turned out to be genuine GPU work (the model's single largest GEMV, batched 4-wide), not removable dispatch overhead.

### VTE vs. llama.cpp/Ollama (reference comparison)

To sanity-check the tok/s figures above against a known-good implementation, both models were benchmarked against Ollama running the *exact same GGUF files on disk* (registered with `ollama create <name> -f Modelfile` using `FROM <absolute-path>`, and cross-checked via Ollama's own reported SHA256 to confirm it wasn't silently re-quantizing or re-downloading anything). Same prompt, same `num_predict`/`max_tokens` (200), `temperature=0` on both sides, decode-only timing (prefill/first-token latency excluded on the VTE side; `eval_count`/`eval_duration` used on the Ollama side, which is the same decode-only window):

| Model | VTE | Ollama (llama.cpp) | VTE as % of Ollama |
|---|---|---|---|
| Qwen2.5-1.5B (Q4_K_M) | 101.68 tok/s (9.84 ms/tok) | 114.67 tok/s (8.72 ms/tok) | 88.7% |
| Granite 4.1 3B (Q8_0) | 45.66 tok/s (21.90 ms/tok) | 51.46 tok/s (19.43 ms/tok) | 88.7% |

The two models have almost nothing in common architecturally (different RoPE convention, different quantization format, ~2x the parameters) and landed on the *same* ~89% ratio to llama.cpp independently. That's a reasonable proxy for "the remaining gap is systemic dispatch/kernel-efficiency overhead across the whole engine, not an architecture-specific bug that happens to be hiding in one model" — llama.cpp has years of GEMV-kernel tuning VTE doesn't have yet, and closing that gap further is a kernel-level efficiency project, not a correctness one. Ollama was used only as a disposable reference during this benchmark — the model, its Modelfile, and the Ollama server process were all removed afterward; VTE has no runtime or build dependency on it.

Two things that were tried and measured as *not* worth it, kept disabled rather than removed, so they don't get re-implemented blindly:
- **FFN kernel fusion** (RMSNorm+Gate+Up+SiLU in one launch): measured slower than the unfused version (12.7–13.5 tok/s vs. 18.8 tok/s) — register pressure from holding two accumulators in one loop drops occupancy enough to erase the round-trip savings. Kept behind `VTE_ENABLE_FFN_FUSION` for anyone who wants to re-run the experiment.
- **WMMA/Tensor Cores**: rejected early for pure batch=1 GEMV (no benefit at that arithmetic intensity). Worth re-measuring now that batch>1 GEMM is a real, validated code path — not yet done.

## How it fits together

Three packages, roughly bottom-up:

**`vte/bridge`** — the only layer that touches the AMD driver directly.
- `hip_runtime.py`: a ctypes wrapper around `amdhip64.dll`. Every allocation is tracked, every pointer is bounds-checked before a memcpy, grid/block/shared-memory dimensions are validated against the real device limits before every launch, and device→host copies are blocked unless the destination tag is explicitly allowlisted (an intentional data-isolation barrier). It also owns two mechanisms that exist specifically because this is a *consumer* GPU shared with the rest of the user's desktop, not a dedicated datacenter accelerator: a `KernelWatchdog` that puts the runtime into a global panic mode if any single launch or `synchronize()` takes too long (protects against a truly hung kernel without doing a risky mid-flight device reset), and a duty-cycle limiter that inserts small preventive pauses before dispatch to keep sustained GPU utilization near 95% instead of pegging it at 100% indefinitely.
- `memory.py`: a slab allocator — one big `hipMalloc` up front, then best-fit sub-allocation with free-list merging inside it, so the engine never pays repeated `hipMalloc`/`hipFree` overhead or fragmentation during a generation loop.
- `gpu_utilization_guard.py`, `kernel_profiler.py`, `watchdog.py`, `dll_discovery.py`, `logger.py`: supporting instrumentation — an independent (Windows performance-counter based) utilization sample used only for diagnostics, on-device GPU-time profiling via HIP events (isolating kernel time from CPU dispatch overhead), the panic-mode watchdog itself, and DLL/toolchain discovery.

**`vte/compiler`** — turns a GGUF file into HIP kernels and a memory layout, ahead of any generation.
- `sanitizer.py` + `gguf_parser.py`: two layers of validation against untrusted `.gguf` input before any tensor is parsed for real (file size/hash/magic/version, tensor bounds).
- `dequantizer.py`: byte-exact Q4_K/Q6_K → FP32/FP16 dequantization matching llama.cpp's reference bit layout — this is where two of the nastier historical bugs lived (see below).
- `weight_loader.py`: mmaps the GGUF and uploads weights to VRAM, either dequantized to FP16 or left raw (Q4_K/Q6_K) for tensors routed to in-kernel dequantization kernels, in host→device chunks small enough to avoid tripping Windows' TDR watchdog on a single giant transfer.
- `qwen_mapper.py` / `granite_mapper.py`: one file per architecture, each computing the full VRAM memory plan (weights, KV cache, activation arena, RoPE cache) and mapping every tensor to an address inside the slab. `qwen_mapper.py` was the single most bug-dense file in the project's early history — KV cache layout mistakes here silently corrupted attention outputs, not crashed — so Granite got its own file rather than growing conditionals into the existing one; see "Multi-architecture support" below for why that mattered in practice.
- `ir.py`, `qwen_compute.py`, `fusion_analyzer.py`, `fusion_applier.py`: build a small IR graph of the model's operations (RMSNorm → MatMul → RoPE → Attention → SwiGLU → …) and look for chains that can be legally fused into a single kernel, given RDNA3 VGPR/LDS limits. Shared by both architectures — Granite's node metadata (`rope_type`, `attention_scale`, `residual_scale`) rides through the same IR rather than needing a parallel graph builder.
- `codegen.py`: renders `.hip.template` files with sanitized parameter substitution (this is the one place where string interpolation into compiled C++ happens, and it's treated as such), invokes `hipcc`, caches binaries by content hash, and rejects any kernel whose VGPR usage risks register spilling.
- `tokenizer.py`: from-scratch tokenizers built straight from each model's own GGUF metadata rather than an external tokenizer library — a byte-level BPE implementation (GPT-2/tiktoken algorithm) for Qwen2.5's vocabulary and merge rules, and a separate `GraniteTokenizer` for Granite's dbrx-style pretokenizer and Jinja2 chat template.

**`vte/core`** — orchestrates a generation request end to end.
- `model.py` (`VTEModel`): the public entry point (`from_pretrained` / `generate` / `generate_batch`). Loads real hyperparameters from GGUF metadata (several early defaults were simply wrong for this model — see bugs below), builds the compute graph, applies kernel fusion, and picks an executor.
- Two executor families, each with a batched variant built by composition rather than inheritance:
  - `fallback_executor.py` / `batched_fallback_executor.py`: eager, node-by-node dispatch through the IR graph. Easier to debug, used as the safety net if graph capture fails.
  - `hip_graph_executor.py` / `batched_hip_graph_executor.py`: captures the entire per-token decode path into a HIP Graph once and replays it, eliminating almost all Python-side dispatch overhead. Static addresses required for graph capture are why the engine pre-allocates persistent activation buffers for every intermediate tensor up front (see "known limitation" below). The LM Head's final GEMV is captured into this same graph (`_capture_lm_head` / `_capture_lm_head_batch`) rather than launched eagerly after each replay, in both the single-sequence and batched executors — `model.py` resolves the LM Head's weight pointer (respecting tied embeddings), logits buffer (`[vocab_size]` or `[batch_size, vocab_size]`), and compiled kernel *before* the respective executor is constructed, specifically to have those fixed addresses ready at capture time. The graph only ever *writes* logits into VRAM — the device→host copy and CPU-side sampling always happen after `graph_launch()` returns, never inside the captured stream.
- `fused_qkv_dispatch.py`: the QKV+RoPE and FFN Gate+Up+SiLU fusion logic shared by both executor families (see "QKV fusion" below for why this exists and what it replaced).
- `kernel_arg_builder.py`: builds the exact `void**` argument arrays HIP kernel launches require, per operation type — this is where a couple of the "silent data corruption" bugs below were actually rooted (an argument count or shape mismatch here doesn't crash, it just feeds a kernel plausible-looking garbage).
- `generator.py`, `sampler.py`, `lm_head.py`: the autoregressive loop (tokenize → prefill → decode-until-stop), CPU-side sampling (repetition penalty, temperature, top-k/top-p, stable softmax — restricted to the top-k candidate subset, not the full vocabulary), and the final hidden-state → logits projection (normally captured into the HIP Graph; `LMHead` keeps an eager fallback path used only when HIP Graph capture fails or is disabled).
- `lifecycle.py`, `motor.py`, `ipc.py`, `gpu_monitor.py`: idle-timeout auto-unload of the model from VRAM, the backend process that the desktop UI talks to over a local pipe, and GPU telemetry (VRAM/temperature/utilization) for the dashboard.

**`vte/ui`** — a Flet desktop app (chat panel + live GPU telemetry dashboard) talking to the `motor.py` backend process over a local pipe.

## Bugs found during development, and what they looked like

Most of the interesting bugs in this project didn't crash — they produced plausible-looking, silently wrong output, which is the worst failure mode for a numerical system like this one. Every one of these was eventually caught by comparing intermediate activations against a NumPy reference implementation layer by layer, not by staring at the code. That's the main reason the `diag_*.py` scripts at the repo root exist as a set of throwaway-but-preserved investigation scripts rather than being deleted after each bug was fixed.

- **Wrong hyperparameters from hardcoded defaults.** Several defaults scattered through early code were simply wrong for Qwen2.5-1.5B: `head_count=16` instead of the real 12, `rope.freq_base=1e4` instead of the model's actual `1e6`, `eps=1e-5` instead of `1e-6`. Wrong head count breaks GQA head-group math and launches attention blocks out of bounds; wrong RoPE base silently rotates embeddings by the wrong angle at every position beyond the first. Fixed by reading every hyperparameter from the GGUF's own metadata instead of a hardcoded table.
- **KV cache V overlapping K.** An early KV cache layout put the V cache in the *second half* of K's allocated space and advanced the per-layer offset using only half the actual size needed. Every layer after the first silently corrupted the layer before it. Fixed by reserving `2x` the per-layer size (K and V distinct) and, once batching was added, multiplying by `batch_size` since each sequence needs its own K/V space from the first token onward.
- **`context_length` metadata mismatch.** The KV cache offset stride must be computed from the *runtime* `context_length` parameter, not the model's native GGUF context metadata (32768 for Qwen2.5, ~16x the old default of 2048). Using the wrong one made the per-layer offset advance faster than the allocated pool, overrunning into the activation arena and subsequent buffers.
- **Input tokens never reaching VRAM.** `_write_input_ids` was, at one point, not actually called before the embedding lookup — the model was processing whatever bytes happened to already be in that buffer, completely ignoring the prompt, and doing so without any error at all.
- **SwiGLU zeroing the FFN.** The SwiGLU kernel takes one `total_elements` argument (`batch * seq_len * intermediate_size`), but the argument builder was passing shape components as three separate arguments instead — the kernel read `total_elements=batch=1` and processed exactly one of the 8960 FFN elements, discarding the rest of the signal on every forward pass.
- **LM head logits reinterpreted at the wrong width.** `matmul_kernel` writes its output in FP16, but the logits buffer was allocated and read back as FP32. The host was reinterpreting pairs of adjacent FP16 values as single FP32 values, producing logits with magnitude ~1e7 instead of the real ~±40 range — with a perfectly correct hidden state feeding into it.
- **Kernel cache key collisions in the HIP Graph path.** Two MATMULs of the same shape but routed to different kernel templates (e.g. a Q4_K `down_proj` via `gemv_q4k` vs. an FP16 `attn_output` via `gemv_coalesced`) hashed to the same cache key when the key was `(op_type, shape)` instead of including the template name. One node would silently inherit the other's compiled kernel and read a quantized weight as if it were raw FP16 — a bug that only manifested in the graph-capture path, since the eager executor happened to key its own cache by name.
- **Embedding lookup silently frozen across graph replays.** The compute graph's `INPUT` node is a marker, always skipped by the dispatcher — `FallbackExecutor` had a manual embedding-lookup step outside the graph loop, but `HIPGraphExecutor` never called an equivalent, so every replay reused whichever embedding happened to be captured once, regardless of the actual token being generated.
- **Q4_K scale/min index bug.** In the sub-block scale/min extraction (`_q4k_scale_min`, replicating llama.cpp's `get_scale_min_k4`), the branch for `j>=4` needs the high 2 bits of `sc` from `scales[j-4]>>6` but the high 2 bits of `m` from `scales[j]>>6` — not `scales[j-4]` for both. Using `j-4` for both happened to work when those particular bits were equal, and silently corrupted part of the dequantized weights when they weren't.
- **Q4_K nibble ordering.** An earlier dequantization pass mixed the low/high nibbles of each byte in 16-element halves instead of the correct 32-element grouping per sub-block, producing weights of plausible magnitude but scrambled values — a model that "ran" and produced fluent-looking garbage.
- **A hardcoded 15ms-per-token floor that became the bottleneck.** A fixed keep-alive delay between decode steps was originally harmless, but after HIP Graph capture and kernel fusion brought the real per-token GPU time well under that floor, the fixed delay became an artificial throughput ceiling (~66 tok/s) that masked the actual gains from every subsequent optimization. Replaced with a much smaller (2ms) keep-alive pulse, just enough to smooth clock-state transitions without becoming the new bottleneck itself.
- **A kernel-argument-count mismatch that looked like a hardware fault.** Two GEMV kernel templates (`gemv_coalesced`, `gemv_q4k`/`gemv_q6k`) take 9 parameters, the last being a `residual_ptr` added later for the epilogue-fusion optimization. A couple of diagnostic scripts were never updated after that signature change and called the kernel with only 8 arguments. Because the runtime's `expected_args` check only validates a caller's own argument-list length against a caller-supplied number — it has no way to know the real compiled kernel's signature — the mismatch went undetected at the Python level, and the kernel read uninitialized memory for the missing parameter: sometimes producing silently wrong output, sometimes dereferencing a garbage pointer and hard-faulting the GPU (a real `VIDEO_ENGINE_TIMEOUT_DETECTED` Windows TDR event). It took ruling out the GPU, the driver, a full OS reboot, and a full ROCm reinstall — all innocent — before isolating it to this one stale call site. The lesson: when a kernel template's parameter list changes, grep for every call site of that kernel; `expected_args` is a self-consistency check, not proof a call site matches the real kernel.
- **Applying the same LM-Head-in-graph fix to batched decode revealed the limits of a naive Amdahl's-law projection.** The same eager-LM-Head pattern existed in `generate_batch()`'s `compute_logits_batch()` call. Measured in isolation, it cost ~7.5ms/tick — the obvious hypothesis was "that's dispatch overhead, capturing it into the batched HIP Graph should recover most of it," projecting end-to-end throughput back up near the ~200 tok/s GPU-only ceiling. After implementing the capture (validated bit-exact against the eager path across all 4 sequences, same as the single-sequence case), the batched decode graph + LM Head fused together measured 24.4ms/tick — barely faster than the 18.5ms (decode) + 7.5ms (eager LM Head) = 26.0ms sum, recovering only ~1.7ms. The correction: most of that 7.5ms was never removable dispatch overhead in the first place — it's the model's single largest GEMV (vocab=151936 × hidden=1536, batched 4-wide, ~933M multiply-adds), and that's genuine GPU work whether it's launched eagerly or captured in a graph. End-to-end batched throughput went from 118.9 to 127.6 tok/s aggregate (+7.4%) — a real, validated win, just a much smaller one than the initial back-of-envelope math suggested. The lesson: a measured cost being "outside the graph" doesn't mean all of it is dispatch overhead — check whether the underlying kernel is cheap (mostly overhead, like the small attention-weight GEMVs elsewhere in this document) or expensive (mostly real work, like this one) before projecting how much a graph capture will actually recover.
- **A measurement protocol that quietly outweighed what it was measuring.** While chasing the 41→100 tok/s gap, an isolated GEMV microbenchmark timed each kernel launch individually (create/record/launch/record/read a HIP event pair, synchronizing after every call — the safe pattern learned from the TDR above). It showed *zero* speedup from routing attention weights through the in-kernel Q4_K/Q6_K dequant kernels, even though the same technique clearly helped the FFN. The reason: at attention's small shapes (K≤1536), the real GPU work is a handful of microseconds, but the per-call host/driver round-trip protocol cost a fixed ~180–200µs regardless of what the kernel actually did — confirmed by timing a trivially small kernel and getting the same ~186µs. The fix wasn't the kernel, it was the ruler: capturing N repetitions of the same kernel into one HIP Graph and timing the whole replay (one round-trip amortized over N real executions) showed the true per-kernel cost was ~6–12µs — and that quantizing attention weights is neutral-to-slightly-negative there, correctly ruling out that whole optimization path with real numbers instead of a noise floor.
- **Forensic profiling of the batch=4 decode graph's 24.3ms/tick contradicted all three a priori hypotheses.** Before spending engineering time on a redesign, the graph was decomposed into its four real components via HIP-Graph-replay-of-N-repetitions timing (the same amortized-overhead technique used elsewhere in this document): QKV fused (13.9%), FlashAttention (1.3%), attention output projection (2.9%), FFN (59.5%), LM Head (22.5%). None of the three suspects that looked most likely on paper — residual Two-Pass Split-K overhead, FlashAttention cache thrashing, or LM Head GEMV-vs-GEMM inefficiency — turned out to be the dominant cost; FFN, not even one of the three, was. This is why the FFN-fusion-at-batch=4 experiment above got prioritized over an LM Head or attention redesign, and why it's worth decomposing a hot loop into real component costs before optimizing the component that *seems* most likely to be slow.
- **The LM Head's low bandwidth efficiency at batch=4 (~7.5%) looked like a block-scheduling locality problem — measured, it wasn't.** Roofline math for the batch=4 LM Head GEMV (reading the ~128MB Q6_K output-embedding weight once per tick): at ~288GB/s theoretical bandwidth, the floor is ~0.44ms; measured in isolation (HIP-Graph-replay timing, `grid=(vocab_size=151936, batch, 1)`, `block=(64,1,1)`) it costs ~1.54ms at batch=1 and ~6.77ms at batch=4 — a 4.39x scaling, worse than the 4.0x of zero cache reuse. The plausible-looking cause: with `blockIdx.x` (the output neuron, 151936 values) as the fastest-varying grid dimension, blocks reading the *same* weight row for different `batch_idx` values are ~151936 block-IDs apart in dispatch order, well past any realistic L2/Infinity Cache retention window. The fix (implemented, numerically validated bit-exact via a dedicated kernel variant, then measured, then reverted): swapping the effective dispatch order so blocks sharing a weight row across the batch launch close together. Two things surfaced along the way worth recording:
  - **`gridDim.y`/`gridDim.z` are capped at 65535 on this hardware** (`hipDeviceProp_t.maxGridSize`, inherited from the CUDA compute-capability convention) — `vocab_size=151936` doesn't fit there, confirmed by a hard `hipErrorInvalidValue` when naively trying `grid=(batch, vocab_size, 1)`. Only `gridDim.x` has room (~2^31-1), so the actual implementation had to flatten to a 1D grid (`blockIdx.x = row*batch + batch_idx`, decoded inside the kernel) to make `batch_idx` the fastest-varying index without violating the limit.
  - **Even after correctly implementing that flattened, ID-adjacent-by-batch dispatch order, the measured result was ~6.56ms — a ~3% improvement, not the projected ~1.8–2.2ms**, and the real `generate_batch()` end-to-end throughput came back statistically flat (120.7 vs. 127.6 tok/s, within run-to-run noise, arguably a wash). **The lesson: block-ID adjacency in a massive grid (607744 blocks across 32 CUs) is not a reliable proxy for temporal adjacency in actual dispatch.** With a kernel this small (64 threads/block), dozens of blocks are resident per CU simultaneously, and the hardware scheduler's real interleaving across 32 concurrently-executing CUs doesn't preserve linear-ID locality tightly enough for an L2 cache line to survive the gap — RDNA3's L2/Infinity Cache behavior under this access pattern can't be reliably steered just by reordering `blockIdx`, at least not at this grid scale. The change (new kernel variant, three call sites) was reverted rather than kept for a statistically-insignificant, unproven gain — this paragraph exists so nobody re-derives and re-implements the same grid swap expecting a locality win that measurement already ruled out.
  - **What the null result actually tells us about the 6.77ms**: it isn't cache thrashing (ruled out above) and it isn't bandwidth (0.44ms theoretical floor, ~15x off). By elimination, the remaining explanation is dispatch/scheduling latency for a genuinely massive kernel — 151936×4 = 607744 blocks is a lot of queue for the GPU's command processor to walk through and feed to 32 CUs, independent of what each block actually does. Closing this further would mean reducing the *number* of blocks (e.g. WMMA/Tensor Cores to fold multiple output neurons per block into a real GEMM tile) or reducing how often the LM Head runs at all (speculative decoding) — both large, separate projects, not a follow-up tweak to this kernel.
- **The actual bottleneck wasn't the GPU at all.** The same graph-replay timing technique, applied to the real production decode graph, showed ~7.6ms/tok of genuine GPU work (a ~130 tok/s ceiling) against a ~13.7–22.8ms/tok wall-clock — meaning something *outside* the captured graph was costing more time than the 28-layer forward pass itself. Decomposing it (timing each piece of the per-token Python loop in isolation) found two culprits: the CPU-side sampler doing `argsort`/`softmax`/`cumsum` over the full 151936-token vocabulary when only the top-k survivors (~50) mattered after filtering (9.4ms, the single largest component measured — larger than the GPU work), and the LM Head's final GEMV running as an eager kernel launch outside the HIP Graph (~3.2ms, paying dispatch overhead the graph exists to eliminate). Fixing the sampler to restrict every downstream operation to the top-k subset before sorting/softmax/sampling cut it to ~0.9ms (bit-exact on the greedy path, statistically identical distribution otherwise — validated before trusting any speed number). Capturing the LM Head into the same decode graph (resolving its tied-embedding weight pointer, logits buffer, and compiled kernel *before* the graph executor is constructed, to have fixed addresses ready at capture time) removed the remaining eager launch. Together: ~41 → ~71.7 tok/s, without touching a single GEMV kernel.
- **A chat UI that never applied a chat template.** The desktop UI sent whatever the user typed straight to `VTEModel.generate()`. Qwen2.5-1.5B-**Instruct** was fine-tuned to expect ChatML markers (`<|im_start|>user\n...<|im_end|>\n<|im_start|>assistant\n`) — without them it just does raw text completion, which is why typing "oi" produced a rambling, unrelated wall of text instead of a reply. Compounding it, `generate()`'s decode loop never checked for the EOS token (`<|im_end|>`), so even a well-formed prompt would keep generating past the natural end of the answer until it hit `max_tokens`, then cut off mid-sentence. Fixed by adding `Tokenizer.apply_chat_template()` (called from the motor process, not the model API itself, so `model.generate()` keeps working standalone for raw-completion use cases) and a stop-on-EOS check in the decode loop.
- **A GUI with no "done" signal, stuck permanently on "generating."** The IPC protocol between the UI and the motor subprocess had messages for tokens, progress, and errors, but nothing for "this generation finished" — so the input field and send button, disabled the moment a prompt was submitted, had no trigger to ever re-enable. It worked by accident during early manual testing because generations were long enough that nobody noticed the UI never recovered after the *last* token. Fixed with a `MotorMsgDone` message sent from the motor's `finally` block (covers natural completion, hitting `max_tokens`, and user-initiated cancellation alike).
- **A cheap "make tok/s feel real-time" idea silently killed the metrics feed.** To show tok/s updating with every streamed token instead of once per second, a metrics snapshot (including a fresh WMI query) was pushed from `flush_tokens()`, which runs on the main generation thread — while a *separate* background thread was already polling WMI every 0.35s for the dashboard. Windows COM objects (what `pythoncom`/`wmi.WMI()` are built on) have thread affinity; touching the same COM connection from two threads doesn't reliably throw where you'd expect, so a naive test looked fine a few times before failing unpredictably. The fix: only the telemetry thread is allowed to touch WMI at all; the per-token push (`_tps_snapshot_msg`) reuses that thread's last full snapshot and only substitutes in fresh tok/s numbers, with `dataclasses.replace`.
- **A live GPU-usage query that made "real-time" dashboard updates 15x slower than intended.** After wiring the dashboard to real WMI performance counters instead of a hardcoded `65.0°C` stub (see below), the telemetry loop's actual update interval on the dev machine was ~5s, not the coded 0.35s. Isolated with per-call timing: `Win32_PerfFormattedData_GPUPerformanceCounters_GPUAdapterMemory` (VRAM) consistently took ~0.03–0.06s, but `...GPUEngine` (GPU usage%, one row per process/engine-type combination system-wide — 228 rows on a normal desktop) took ~3.3–4.4s *every single call*, not just the first. The usage% it produced was never even wired into the UI. Removed entirely — it was pure cost for a number nobody saw, and it was single-handedly throttling every other metric down to its own pace.
- **A GPU dashboard that turned out to be entirely fake.** `GPUMonitor._query_wmi_windows()` ran a WMI query, discarded the result, and returned hardcoded constants (`temperature: 65.0`, `gpu_usage: 45.0`, `fan_speed: 1200.0`) unconditionally — a stub that was never filled in, indistinguishable at a glance from real telemetry. Replaced VRAM and usage with real WMI performance-counter data (`GPUAdapterMemory`, the same source Task Manager's GPU tab reads from). Separately, `WMI>=1.5.10` in `pyproject.toml`/`requirements.txt` pinned a version that **does not exist on PyPI** (1.5.1 is the latest) — meaning a fresh `pip install` of this project's declared dependencies had been silently unsatisfiable the whole time; nobody had hit it yet because the dev environment's `WMI` package predated the pin, or was never installed at all (in which case the code's own `except ImportError`-shaped fallback masked it). Fixed the pin and added the missing `pywin32` (provides `pythoncom`, required by the `wmi` package). Real GPU temperature specifically wasn't fixable via WMI — `Win32_VideoController`, `MSAcpi_ThermalZoneTemperature`, and the LibreHardwareMonitor/OpenHardwareMonitor WMI namespaces were all checked and none exposed it — see the ADL entry below for how it was eventually solved.
- **Verify an SDK's enum values against the real header before trusting them, not from memory.** AMD's ADL (Display Library, `atiadlxx.dll`, ships with every AMD driver) can read real GPU temperature, but the specific numeric sensor IDs (which integer means "edge temperature" vs. a clock speed) live in `adl_defines.h` and change between API generations — guessing one would silently read the wrong sensor and label it as temperature, exactly the "plausible but wrong" failure mode this document keeps returning to. Fetched the real values from AMD's public SDK repo before writing any ctypes binding (`ADL_PMLOG_TEMPERATURE_EDGE = 8`, confirmed against a working third-party reference implementation for the exact array-indexing convention and value scaling). Two things the docs didn't mention, only caught by testing against the real card: `AdapterInfo.iVendorID` reports AMD's PCI vendor ID (`0x1002`) as the *decimal digits* `1002` interpreted as an integer (`0x3ea`), not the hex value itself — checking only `0x1002` would have matched zero adapters; and the legacy `ADL_Overdrive5_Temperature_Get` API (pre-RDNA, GCN-era) returns a clean explicit error code on this RDNA3 card rather than silently succeeding, which is what made it safe to try it as a fallback without risking a wrong-but-plausible reading. See [vte/bridge/adl_bridge.py](vte/bridge/adl_bridge.py).
- **A model path that only worked by accident of always being launched from the same directory.** `VTEModel.from_pretrained()` resolved the GGUF as `Path("Model") / filename` — relative to whatever the process's current working directory happened to be at launch, not to the project itself. It worked throughout development because the dev shell's cwd was always the repo root, and failed immediately ("model file not found") the moment the desktop UI's installed `vte-ui` command was run from anywhere else — despite the file being exactly where it should be. Fixed by checking both the cwd-relative path (for anyone who wants their own `Model/` elsewhere) and a path resolved from `__file__` back to the repo root, and printing both full paths that were checked when neither has the file.
- **Throughput degrading over the course of a single long generation, traced to 20 of 32 CUs sitting idle during attention.** A user-reported symptom ("short replies stay fast, long stories slow down") led to isolating `flash_attention_kernel`'s cost at different KV-cache positions via HIP-Graph-replay timing (one layer, in isolation): 6.9µs at position 1 growing to 211µs at position 300. First question: is this O(N) (expected — causal attention must scan more history per token) or O(N²) (a tiling/LDS bug)? Checked by comparing costs at exact 2x position ratios (100→200, 150→300) instead of naively multiplying: both gave ~1.9x, not ~4x — confirming clean O(N), not quadratic. But a roofline check on that O(N) slope (~0.7µs per additional KV position, 28 layers) showed it was ~800x worse than the pure memory-bandwidth floor for reading that much additional K/V data — meaning the *shape* of the growth was correct but the *constant* was wildly inflated. Root cause: `flash_attention_kernel` launches `grid=(batch, num_q_heads, 1)` — only 12 blocks at batch=1, leaving 20 of the RX 7600's 32 CUs idle regardless of context length — and the per-KV-position score computation inside each block is a fully serial loop, so all the growth lands on those same 12 blocks with no added parallelism to absorb it.
  - **The fix (`flash_attention_split_kv_*.hip.template`, default-on, opt-out via `VTE_DISABLE_ATTN_SPLITKV=1`) mirrors the QKV Two-Pass Split-K precedent, applied to the KV-sequence dimension instead of the output dimension.** Three kernels replace the original one: `kv_cache_append_kernel` writes the current token's K/V once (pulled out of the attention kernel specifically so it doesn't need per-chunk boundary guards); `flash_attention_split_kv_partial_kernel` splits the KV cache into 32-position chunks, one block per (batch, q_head, chunk) — at position 250 this raises the grid from 12 to 12×8=96 blocks — each computing local online-softmax accumulators (`m`, `l`, `o`) into a scratchpad; `flash_attention_split_kv_reduce_kernel` merges the (up to `context_length/32`) partial results with the standard online-softmax correction (`exp(m_local - m_global)`), caching that correction in LDS once per chunk instead of recomputing it per output dimension. The scratchpad (`~390KB` for `context_length=2048`) is allocated once via the same `SlabAllocator` pattern as every other persistent buffer, sized for the full configured `context_length` up front — required because grid dimensions are fixed at HIP Graph capture time and can't shrink as the KV cache actually grows, so most chunk-blocks are harmless early-exit no-ops during the first ~30-40 tokens of any generation.
  - **"Bit-exact" was the wrong bar to validate against, and checking it the naive way (compare free-running greedy output end to end) initially gave a false alarm.** Short generations (≤80 tokens) matched token-for-token with the feature on vs. off, but a 140-token generation diverged partway through — different word choice from that point on. That's not automatically a bug: splitting the KV cache into chunks and merging the partial online-softmax results is a *different order of floating-point operations* than one long serial accumulation, and reordered-but-mathematically-equivalent float ops are not generally bit-identical (the same non-reproducibility shows up comparing different compiler flags or reduction orders on *any* numerical code). The right test is whether the logits themselves agree within FP16/FP32 rounding tolerance at the same context, not whether free-running greedy decoding never takes a different path. Isolated that properly: loaded two model instances, fed both the *identical* token sequence (the original's own greedy output, replayed into both rather than letting each generate independently) for 150 decode steps, and compared raw logits at every step. Max logit difference stayed in the ~0.02–0.07 range (on logits spanning roughly ±40) at every single position, and the argmax token never flipped once across all 150 steps. The 140-token divergence was a real event, just not a bug: at that specific position the top-2 logits were apparently closer together than that ~0.07 noise floor, so the tiny reordering difference flipped which one won a near-tie — after which autoregression means everything downstream legitimately diverges, correctly, from a different chosen word. This has no practical effect at the project's actual default (`temperature=0.7`, stochastic sampling) — noise at that scale is negligible next to intentional sampling randomness — and only matters at all for exact-reproducibility of greedy (`temperature=0`) decoding specifically.
  - **Isolated per-layer measurement revealed a real trade-off, not a pure win**: below roughly position 35, the original single-kernel design is still faster (e.g. 6.9µs vs. 17.8µs at position 1) — launching ~768 mostly-empty chunk-blocks costs more than the serial loop it replaces at short context. Beyond ~50, Split-KV's cost flattens to a near-constant ~39µs regardless of position, vs. the original's unbounded growth (a ~5.4x win by position 300).
  - **The real end-to-end number is what mattered, and it was better than the isolated numbers predicted**: measuring the full 28-layer decode graph (not just one layer) across the same 250-token generation, the short-context regime showed *no* measurable regression — throughput held at a flat ~78–79 tok/s from the first token to the last, actually *higher* than the original design's first-25-token average (70.6 tok/s), not just its declining tail (50.8 tok/s by the end). The likely explanation: at short context, FFN and QKV already dominate the tick, so the ~0.3ms/tick difference (28 layers × ~11µs) that showed up in isolation gets absorbed into noise at the full-graph level.
- **A hardware safety guard that outlived its own justification, and one intermediate value that was quietly the most dangerous.** After Split-KV cut the decode tick to ~9-10ms, the fixed 2ms keep-alive pulse between tokens (added earlier specifically to stop the WDDM driver's power management, DPM, from dropping GPU clocks between bursts and risking stutter or a TDR) started costing a disproportionate ~15-20% of every tick. Before touching the production code, the exact same "measure before deciding" discipline used throughout this document was applied: a standalone benchmark script parametrized the pulse duration (2.0, 1.0, 0.5, 0.1, 0.0ms) and measured both average tok/s and per-token variance (a stutter proxy — DPM throttling should show up as the back half of a long generation getting slower and jumpier than the front half) across a continuous 200-token generation *and* four separate conversation turns each preceded by a real 3-second idle gap (the actual scenario the pulse was meant to protect). The result inverted the expected risk direction: 0.0ms (fully disabled) was both the *fastest* (102 tok/s vs. 81.8 tok/s at 2.0ms in the isolated benchmark; 100 tok/s stable end-to-end in production) and the *most stable* (0.109ms stdev over the last 50 tokens, front-half/back-half difference of +4% — if anything the second half got slightly faster) of every value tested, with zero TDR across either test shape. The real danger turned out to be hiding at **1.0ms**, a value nobody would have picked as risky by intuition: 213.5% swing in the last-50-tokens window, with individual token latencies spiking to 35ms (~3x the average) — a genuine resonance/instability point between the pulse's own timing and the decode loop's cadence, worse than either the old default (2.0ms) or full removal. The physical explanation for why 0.0ms works now: WDDM's DPM takes on the order of hundreds of milliseconds to decide to downclock, and a decode loop now dispatching a kernel every ~10ms never leaves a gap long enough for DPM to act — the artificial pacemaker was solving a problem that a faster natural heartbeat had already made obsolete. Shipped as the new default with an escape hatch, not a deletion: `VTE_KEEPALIVE_PULSE_MS` (default `"0.0"`) lets the pulse be restored without a code change if a future driver update, different hardware, or contention from another GPU-heavy application ever reintroduces the original stutter risk. The general lesson: a safety guard's correct value is a function of the performance baseline it was tuned against — re-measure it (all the way through, including "obviously safe" intermediate values, not just the two extremes) whenever that baseline moves, since the guard can go from necessary, to irrelevant, to actively the worst option in the range, depending on what else changed around it.

## Multi-architecture support: adding Granite 4.1 3B

Granite 4.1 3B (Q8_0) was added as a deliberate stress test of the codebase's assumptions, all of which had been built and validated against exactly one model. It shares the compiler/IR/executor pipeline with Qwen almost entirely — new code was needed only where the architectures genuinely differ: `granite_mapper.py` (memory plan), `GraniteTokenizer` (vocabulary/chat template), a few extra per-node metadata fields (`rope_type`, `attention_scale`, `residual_scale`) threaded through the existing IR, and a raw-Q8_0 dequant path in the compiler alongside the existing Q4_K/Q6_K one. `SUPPORTED_ARCHITECTURES` (in `sanitizer.py`) and `VTEModel.MODEL_REGISTRY` (in `model.py`) are the two tables that make a new architecture selectable at all — both list Qwen2.5 and Granite side by side, and `model_name=` at `from_pretrained()`/CLI/UI-dropdown time is what picks between them (see "Getting started").

Two of the bugs found while bringing Granite up were serious enough to cover in detail here rather than as one-line entries above, because both are the kind of "plausible but wrong" numerical bug this document keeps coming back to — the model ran, produced fluent-looking text, and was still wrong:

- **RoPE rotation convention mismatch (NEOX vs. NORM) — the actual root cause of Granite's incoherent, mixed-language output.** VTE's shared RoPE-applying kernels (`split_k_qkv_pass2`, `fused_norm_matmul_rope`, `rope.hip.template`) only ever implemented one rotation convention: split-half pairing, where dimension `j` of a head pairs with `j + head_dim/2` to form a rotation pair. That's correct for Qwen2.5 — but not universal. Checked against llama.cpp's actual source (`llama_model_rope_type()` in `llama-model.cpp`, not guessed or assumed from a paper) confirmed `LLM_ARCH_QWEN2` maps to `LLAMA_ROPE_TYPE_NEOX` (split-half) while `LLM_ARCH_GRANITE` maps to `LLAMA_ROPE_TYPE_NORM` (interleaved pairing: dimension `2i` pairs with `2i+1`) — two genuinely different conventions, not a parameterization of the same one. Running Granite through the NEOX-only kernels rotated every embedding by the wrong angle at every position beyond the first, and the result looked like a plausible language-model bug (garbled, mixed-language text) rather than an obviously-broken one — exactly the failure mode that makes bugs like this expensive to find without a reference to diff against. Fixed by threading a `rope_type` parameter (0=NEOX, 1=NORM, read from the same per-architecture metadata table as everything else) through all three kernel templates and their Python argument builders, branching to the correct pairing at kernel level. Granite went from garbled output to fully coherent, correct answers immediately after this one fix.
- **`residual_scale` applied to every projection instead of only the ones feeding a residual add.** Granite defines a `residual_scale=0.22` (a value Qwen doesn't have at all) meant to scale a sublayer's output immediately before it's added back into the residual stream — confirmed against `granite.cpp`'s own source (`ggml_scale(cur, f_residual_scale)` called right before `ggml_add(cur, inpSA)` / `ggml_add(cur, ffn_inp)`, and nowhere else). VTE's kernel-argument builder instead applied it unconditionally to every MATMUL node's output, including `gate_proj`/`up_proj`/`q_proj`/`k_proj`/`v_proj` — tensors with no residual add downstream at all. Fixed by gating the real scale value behind the existing residual-fusion lookup (`RESIDUAL_FUSION.get(node.name)`, already used elsewhere to decide which nodes get epilogue-fused with their residual add), falling back to `1.0` for every other node. First-token confidence on a held-out prompt went from ~11% to ~57% after the fix — a large, measurable jump from what looked like a small scoping mistake.

Two smaller, more mundane findings from the same effort, worth a line each so they aren't silently rediscovered:

- **Raw Q8_0 dequantization only pays off for some tensor shapes, not all of them — measured, not assumed.** Q8_0's block layout (34 bytes / 32 elements) isn't a multiple of 16 bytes, so `gemv_q8_0.hip.template` can't use the same `uint4`-vectorized block reads `gemv_coalesced`/`gemv_q4k` rely on; its per-block unpack loop is scalar. Routing `attn_output` (a smaller GEMV shape) through raw Q8_0 as an experiment measured *slower* than dequantizing to FP16 first and using the existing `gemv_coalesced` kernel (~32.4–36 tok/s vs. a ~36.3+ tok/s FP16 baseline) — the scalar unpack overhead outweighs the bandwidth saved at that shape. `granite_mapper.py::is_raw_q8_0_weight()` excludes all four `attn_*` tensors (`attn_q`, `attn_k`, `attn_v`, `attn_output`) from raw-Q8_0 routing for this reason; only the larger `ffn_gate`/`ffn_up`/`ffn_down`/`token_embd` tensors stay raw, where the bandwidth savings do win.
- **`FallbackExecutor.num_layers` was hardcoded to `28`** (Qwen's actual layer count) instead of being read from model metadata — found incidentally while profiling Granite in eager mode (the profiler's launch counts were suspiciously stuck at a 28-layer-consistent pattern for a model that has 40 layers). Low real-world impact, since the default HIP Graph path was unaffected and `FallbackExecutor` is only the safety net used if graph capture fails — but a genuine silent-truncation bug in that fallback path nonetheless. Fixed to `self.metadata.get('block_count', 28)`.

Once correctness was established, Granite got the same "measure, don't guess" throughput treatment as Qwen (see "Performance"): vectorizing the Split-K QKV Pass1 kernel's inner dot-product loop with `__half2` (pairing two `__half` reads/multiplies per iteration instead of one — safe here because `chunk_size = hidden_size/32` is even for both models, 48 for Qwen and 80 for Granite) delivered a measured ~10% steady-state improvement (~42–44 → ~45–48 tok/s). Granite fits in ~4.8GB VRAM including the SlabAllocator's margin (measured via `hipMemGetInfo`, the real system-wide GPU memory counter, not just VTE's own internally-tracked sub-allocation sum — see "Bugs found during development" region above for why those two numbers differ) and reaches the same ~89% of llama.cpp/Ollama's throughput that Qwen does (see "VTE vs. llama.cpp/Ollama" above) — real evidence that the RoPE/residual-scale fixes above generalize the architecture rather than papering over one model's specific quirks.

### The Flet UI's model-switch race conditions

Letting the desktop UI switch models at runtime (not just at startup) surfaced two related concurrency bugs in `Orchestrator`/`VTEApp` — both real, both reported by direct testing rather than by inspection, and both fixed with the same underlying idea: a monotonically increasing "generation" counter that tags which motor subprocess a message actually came from.

- **A false-positive "motor pipe closed unexpectedly" error on every single model switch**, not an occasional race — this was reported explicitly ("isso sempre acontece") after an initial hypothesis that it was a rare startup-timing race turned out to be wrong. Root cause: `Orchestrator._listener_generation` was incremented *after* `self.motor_process.join(timeout=...)` returned for the old process, but the old `_listen_pipe` background thread's `EOFError` (from that same old pipe legitimately closing during the join wait) could fire while the generation counter still matched — so a perfectly normal shutdown got reported as an unexpected error. Fixed by moving the increment to the very top of `start_motor()`, before the old process is touched at all.
- **Stale messages from an already-replaced motor generation leaking into the new model's session** — not just the error above, but any message, including a legitimate `MotorMsgDone` from the old process arriving after the new one had already started loading. Symptom, reported directly: sending a message to Granite while it was still loading (nothing prevented it) could re-enable the input field via a straggling old-generation "done" message before the new model was actually ready, and in one case produced an unhandled error just from pressing Enter. Fixed two ways together: `_listen_pipe` now re-checks `generation == self._listener_generation` immediately before forwarding *every* message (not just the final error), and a new `model_ready` boolean on `VTEApp` — set `False` on every `reset_for_new_motor()`, `True` only when a current-generation `MotorMsgReady` arrives — independently gates whether `_handle_submit` will send anything at all. `pipe_conn.send()` calls in the UI's event handlers were also wrapped in try/except (`BrokenPipeError`/`OSError`/`EOFError`), since a send racing a subprocess restart can legitimately fail even with both fixes above in place.
- **A visually-alarming crash banner during this same testing turned out not to be a real GPU crash.** A screenshot showed what looked like a WDDM TDR event ("Núcleo da GPU abortado"). Investigated against ground truth rather than trusting the banner's own text: Windows Event Log's System channel (Event ID 4101, the actual TDR event) and Application channel (Event ID 1000, Windows Error Reporting crash events) via `Get-WinEvent -FilterHashtable`, plus checking for a fresh `KernelWatchdog` timeout file. None of the three fired. The banner was a downstream symptom of the same generation-race bug above (a motor process being `.terminate()`d mid-boot during a rapid model switch, surfaced by the UI as an alarming-looking generic error), not a real driver/GPU fault — confirmed once the race itself was fixed and the same test sequence stopped producing the banner at all.
- **A hidden `max_tokens=512` default silently capped every generation, masking two symptoms as one bug report.** `UIMsgPrompt.max_tokens` defaults to `512` in `ipc.py`, and `_handle_submit` never overrode it with the UI's own context-length dropdown value — so long generations got cut off mid-word once they crossed 512 tokens (reported directly, with the actual cut-off text pasted in: a story ending mid-word, "...contra inv"), and separately, the context-length dropdown *looked* broken because changing it had no visible effect on anything (`current_seq_len >= self._context_length` in the decode loop was already correct and unrelated — the cap was hit long before that ceiling could matter). Fixed by sending `max_tokens=self.current_context_length` instead of relying on the dataclass default. Confirmed with a real generation at `context_length=1024`: it now completes naturally (ends in a period, sends `MotorMsgDone(cancelled=False)`) instead of being cut mid-word at 512.
- **Light theme left chat bubbles on their dark-theme colors.** `_make_bubble()` baked the current palette's colors into each bubble at creation time; the theme-toggle handler (`_apply_palette()`) only re-themed the UI chrome, never revisited bubbles already on screen. Fixed by tracking every created bubble (`self.chat_bubbles`) and re-theming each one's background, border, and text colors inside `_apply_palette()`.
- **A generic "loading" screen was reused for model switches, which could be mistaken for the UI having frozen** rather than genuinely loading a different (larger) model — flagged directly as a UX concern, not a bug. `reset_for_new_motor()` now distinguishes initial boot, same-model restart, and model-switch (via a `model_name != self.current_model_name` check made *before* overwriting the tracked name) and shows a distinct title/status message for each.

## Why QKV projection is fused, and why FFN fusion is off

These two decisions look inconsistent at a glance (fuse one, not the other) but both came from measurement, not intuition:

The original QKV design launched one kernel block per attention head (12 heads for Q, 2 for K/V) — on a 32-CU GPU, that leaves most compute units idle (occupancy-bound, not ALU-bound). The current design (Two-Pass Split-K) does a single shared RMSNorm, then splits each projection's inner dimension across 32 blocks regardless of head count, filling every CU. Measured: 37.9 → 41.0 tok/s (+8%), validated numerically and end-to-end with real text.

FFN fusion (RMSNorm+Gate+Up+SiLU as one launch) was built the same way and measured *slower*: 18.8 tok/s unfused vs. 12.7–13.5 tok/s fused. The FFN's grid only has ~35 blocks (`intermediate_size / 256`), and holding two running accumulators (gate and up) in the same loop increases per-thread register pressure enough to drop occupancy below the point where fewer VRAM round-trips pay for themselves. It's kept in the codebase, disabled by default, behind `VTE_ENABLE_FFN_FUSION`, specifically so this negative result doesn't get silently rediscovered and re-implemented by someone who didn't see the measurement.

**Technical debt: `fused_gate_up_silu_kernel` is incompatible with the project's current FFN weight format, independent of the batch=1 regression above.** While extending this template to support `batch>1` (in case a batch=4 regime, with more real GPU work per launch, changed the occupancy math enough to flip the fusion decision — it didn't, see below), the kernel produced NaN output at both batch=1 and batch=4. Root-caused by bisecting against `git stash` (i.e. confirming the *unmodified, original* template also produces NaN, proving this predates and is unrelated to the batch-support edit): `fused_gate_up_silu_kernel` reads `gate_weight`/`up_weight` through a `const __half*` cast, but `blk.N.ffn_gate.weight`/`blk.N.ffn_up.weight` are actually stored raw in Q4_K format (~4.5 bits/element) — the project switched FFN weights to raw in-kernel Q4_K dequant (via `gemv_q4k`) at some point after this fusion template was written, and the template was never updated to match. It has been dormant and silently broken the entire time only because `VTE_ENABLE_FFN_FUSION` defaults to off. Properly fixing it means adding Q4_K block unpacking (scale/min extraction, 4-bit unpack) *inside* the same loop that already holds two running accumulators — deliberately rejected (Option 1) rather than attempted, because it very likely worsens the exact register-pressure problem that already made this fusion a net loss at batch=1: Q4_K unpacking needs its own working registers (block scale, min, 4-bit nibble buffer) on top of the gate/up accumulators already present, on a kernel that already loses to VRAM round-trips at *lower* register pressure than that. Decision: do not attempt to fuse the FFN again until the project's FFN weight quantization format changes (e.g. if a future pre-dequantized-to-FP16 weight path is introduced) — re-deriving this from scratch instead of reading this note first would waste real debugging time on a NaN that isn't caused by whatever new code triggered it.

## Security posture

The only input to this project that should be treated as genuinely untrusted is a `.gguf` file, since these are commonly downloaded from third-party sources. It goes through two validation layers (`GGUFSanitizer` then `GGUFParser`) before any tensor data is used: file size/hash/magic/version checks, tensor count/KV count caps, and per-tensor offset/size bounds checking against the actual file. Two things that look like gaps but are intentional, documented design decisions: the GPU utilization guard is observation-only by design (it never kills in-flight work), and the UI↔motor IPC pipe has no authentication (local-only threat model, not designed to cross a network boundary).

## Notes for anyone reviewing this from an AMD/ROCm perspective

A few things in this codebase exist specifically because of gaps or friction encountered targeting a consumer RDNA3 part through HIP on Windows, rather than a datacenter part through the usual ROCm software stack. Flagging them here in case any have a better-known solution, or are useful data points either way:

- **`hipcc` doesn't find MSVC/Windows SDK headers on its own on Windows** — `codegen.py`'s `_setup_hip_env` manually injects `PATH`/`INCLUDE` entries before invoking it, or compilation fails with `cmath not found`. This felt like something that should be handled by the SDK installer or `hipcc` itself.
- **A single large synchronous `hipMemcpy` (Host→Device) is a practical TDR trigger on Windows/WDDM.** Uploading `token_embd.weight` (~445MB) in one call was enough to risk the driver considering the GPU hung. `weight_loader.py` chunks all host→device uploads into ≤16MB pieces specifically to give the driver windows between calls. It's not clear whether this is a WDDM constraint that a well-behaved HIP application on Windows is just expected to work around, or something the driver could handle more gracefully.
- **No official signal for "this consumer GPU is a shared desktop resource, keep it under X% utilization sustained."** `HIPRuntime`'s duty-cycle limiter (`_throttle_before_dispatch` / `_enforce_duty_cycle_limit`) is a from-scratch mechanism that measures real busy-time in a sliding window and inserts pauses to keep sustained utilization near 95%, specifically so the engine doesn't make the rest of the user's desktop unresponsive during long generations. There's no ROCm/driver-level equivalent being used here — this is pure userspace timing.
- **VGPR/register-spill risk is checked by parsing `hipcc`'s own compiler output** (`codegen.py::_parse_vgpr_usage`), not queried through any structured API — kernels whose reported VGPR usage exceeds 128 are rejected before ever being launched. A structured way to query this (occupancy calculator equivalent for arbitrary generated kernels) would remove some fragility here.
- **The FFN fusion regression (above) is architecture-specific to RDNA3's 32 CUs at this problem size** (`intermediate_size=8960` → ~35 blocks). It's plausible this fusion would actually win on a part with a very different CU count/register file ratio; the measurement here should be read as "wrong for this specific GPU," not as a general claim about the fusion technique.
- **HIP Graphs eliminate CPU-side dispatch overhead but not GPU-side kernel-boundary cost** (weight-streaming interruption + restart latency + intermediate HBM round-trips between graph nodes). There's an open, not-yet-measured question about how large that boundary cost actually is on a 32-CU RDNA3 part, and whether fusing a few more adjacent stages would be worth it here specifically.
- **A single-sequence decode step still uses only a fraction of the RX 7600's theoretical memory bandwidth — but the gap is now isolated to the GEMV kernels themselves, not overhead around them.** Rough math: at ~288 GB/s theoretical GDDR6 bandwidth and ~1.14GB of weights read per token, the physical floor is ~4ms/token (~250 tok/s ceiling). Measured via HIP Graph replay (timing N repeated replays of the real decode graph, isolating GPU time from any host-side cost), the 28-layer decode graph itself runs at ~7.6ms/tok — a ~130 tok/s ceiling, about 52% of the theoretical floor. End-to-end throughput (~100 tok/s, stable across generation length) is now close enough to that GPU-only ceiling that the remaining gap is mostly the graph's own kernels, not dispatch, occupancy, or CPU-side overhead (all found and fixed — see "Bugs found during development"). Closing the ~130 vs ~250 gap further would mean comparing kernel-level design choices (block size, load pattern, LDS usage) against other HIP/ROCm GEMV implementations on the same hardware — not yet done; any insight into what a well-tuned RDNA3 GEMV should be capable of in practice (vs. datasheet bandwidth) would be useful context.

## Known limitations

- `generate_batch` requires every prompt in a batch to have identical token length (lockstep only); padding + attention masking for mixed-length batches is unimplemented.
- Batch sizes above 4 currently regress throughput (see the performance table above) — not yet root-caused beyond the Infinity Cache hypothesis.
- Persistent activation buffers used by the HIP Graph executor are sized for `seq_len=1` (the dominant decode case); a multi-token prefill through that executor could in principle overrun those buffers. The current code avoids this by processing prefill token-by-token even in HIP Graph mode, reusing the single decode graph — but the buffers themselves aren't yet partitioned by shape class to make this impossible by construction rather than by calling convention.
- FFN kernel fusion and WMMA/Tensor Cores remain measured-and-rejected (former) or unevaluated in the current batch>1 regime (latter) — see the performance section.
- `fused_gate_up_silu_kernel` (`VTE_ENABLE_FFN_FUSION`, off by default) is currently broken (produces NaN) for the project's actual FFN weight format — it assumes FP16 weights but `ffn_gate`/`ffn_up` are stored raw Q4_K. This predates and is unrelated to the batch=1 performance regression that already kept this fusion disabled. See "Why QKV projection is fused, and why FFN fusion is off" for the full root-cause and why fixing it in-kernel was deliberately rejected rather than attempted.
- The batched (batch=4) LM Head GEMV achieves only ~7.5% of theoretical memory bandwidth, notably worse than the FFN's ~25%. A grid-dispatch-order fix (making batch_idx the fastest-varying block index) was implemented, validated bit-exact, measured (~3% gain, statistically flat end-to-end), and reverted — see "Bugs found during development" for the full measurement. Remaining ~15x gap vs. the bandwidth floor is attributed to per-block dispatch/scheduling latency across 607744 blocks, not cache locality; closing it further needs WMMA/Tensor Cores or speculative decoding, both out of scope for a kernel-level tweak.
- Single-sequence decode throughput (~100 tok/s, stable across generation length) is still below the GPU's theoretical memory-bandwidth ceiling (~250 tok/s) and the decode graph's own measured GPU-only ceiling (~130 tok/s) — the CPU/dispatch overhead, attention occupancy, and artificial pacing costs that used to dominate this gap have been found and fixed (sampler, LM Head, Split-KV, keep-alive pulse), so what remains is genuinely about GEMV kernel efficiency, not overhead around it. Not yet closed.
- `LMHead`'s HIP-Graph capture path assumes the decode graph always runs at `seq_len=1` (true for the current lockstep decode/prefill scheme) — a future multi-token-per-step scheme would need the LM Head capture logic revisited alongside it.
- Real GPU temperature on Windows only works on AMD hardware, and only if either the PMLog (`ADL2_New_QueryPMLogData_Get`, RDNA/RDNA2/RDNA3) or the legacy Overdrive5 (`ADL_Overdrive5_Temperature_Get`, pre-RDNA) API responds — see [vte/bridge/adl_bridge.py](vte/bridge/adl_bridge.py). On a non-AMD GPU, or if both ADL calls fail, `GPUMonitor` reports `None` and the UI shows "N/A" rather than fabricate a number. VRAM and the model's idle-unload countdown are WMI-sourced and always real, refreshed every ~0.35s.
- The dashboard shows two separate VRAM numbers, deliberately not one: the large value is what VTE itself allocated (weights+KV+arena, deterministic, always equals the Weights/KV/Arena breakdown below it); "Sistema (dedicada)" is the whole system's dedicated GPU memory from WMI (matches Task Manager's GPU tab, but includes the desktop compositor, browser, etc.). An earlier version conflated the two into one number, which made it jump in ways that had nothing to do with VTE's own usage.
- Split-KV attention (see "Bugs found during development") is the default for single-sequence decode (opt-out via `VTE_DISABLE_ATTN_SPLITKV=1`) after a validation pass covering short/long generations, chunk-boundary crossings, mid-generation cancellation, and scratchpad reuse immediately after a cancellation — all produced correct, coherent output with `pytest` green throughout. It only affects `generate()`; `generate_batch()` (batch>1) uses the unmodified `flash_attention_kernel` and is untouched by this change. One caveat worth knowing: greedy (`temperature=0`) output can occasionally diverge from the pre-Split-KV baseline partway through a long generation — not a bug, see the bug entry for why (reordered-but-equivalent floating point, not incorrect floating point), and irrelevant at the project's actual default (`temperature=0.7`).
- The inter-token keep-alive pulse now defaults to `0.0` (disabled) — see "Bugs found during development" for the measurement showing this is currently both faster and more stable than any nonzero value tested on this hardware/driver. `VTE_KEEPALIVE_PULSE_MS` (milliseconds, e.g. `"2.0"`) restores the old behavior without a code change if a future driver update, different hardware, or another GPU-heavy application running alongside VTE ever reintroduces the DPM-downclocking stutter this was originally built to prevent.

## Getting started: the Python API and desktop UI in detail

```python
from vte.core.model import VTEModel

model = VTEModel.from_pretrained("qwen2.5:1.5b-q4_k_m", context_length=8192)

for token in model.generate("Summarize the tradeoffs of static batching:", max_tokens=300):
    print(token, end="", flush=True)
```

Switching architectures is just a different `model_name` — everything else about the API is identical (see "Multi-architecture support" above for what actually changes under the hood):
```python
model = VTEModel.from_pretrained("granite-4.1:3b-q8_0", context_length=8192)

for token in model.generate("Summarize the tradeoffs of static batching:", max_tokens=300):
    print(token, end="", flush=True)
```

Batched decode (same-length prompts only, see "Known limitations"):
```python
model = VTEModel.from_pretrained("qwen2.5:1.5b-q4_k_m", max_batch_size=4)

prompts = ["The capital of France is", "2 + 2 equals", "My favorite color is", "Once upon a time"]
for words in model.generate_batch(prompts, max_tokens=200, temperature=0.7):
    print(words)  # one word per sequence, per generation tick
```

### Desktop UI

The UI (chat + live dashboard: tok/s, ms/token, VRAM, model lifecycle, logs) lives entirely in [vte/ui/app.py](vte/ui/app.py), a Flet 0.85+ app. It needs `vte` to be an *importable package*, not just a folder of scripts — running the file directly with a system-wide Python (`python vte/ui/app.py`) fails with `ModuleNotFoundError: No module named 'vte'`, because that Python has never heard of this project.

**Setup (once):**
```bash
python -m venv .venv
.venv\Scripts\activate          # Windows
pip install -e .[dev]
```
This registers `vte` as an editable package in `.venv` — `import vte` and the `vte-ui` command both work from anywhere afterward, without needing to run scripts from a specific directory.

**Run it** (with `.venv` activated, and a `Model/` directory containing the GGUF):
```bash
vte-ui
```
Equivalent, if you don't want to rely on the installed console script:
```bash
python -m vte.ui.app
```
Note the `-m` — `python vte/ui/app.py` (running the file by path) still fails even with `.venv` activated, for the reason above; `-m` runs it *as a module* inside the `vte` package, which is what makes the relative imports resolve.

**Windows-only telemetry dependencies** (`WMI`, `pywin32`, both in `pyproject.toml`) are what let the dashboard read real GPU numbers (dedicated VRAM usage, matching what Task Manager's GPU tab shows) instead of placeholders. Without them the dashboard still works, just with less data. Real GPU temperature needs no extra package — it goes through AMD's ADL (`atiadlxx.dll`, ships with every AMD driver) via [vte/bridge/adl_bridge.py](vte/bridge/adl_bridge.py) — but only works on an AMD GPU; on anything else (or if ADL fails for any reason) the dashboard shows "N/A" rather than a fabricated number.

The GGUF itself goes in a `Model/` folder — `VTEModel.from_pretrained()` checks it both relative to the current working directory and relative to the repo root, so it's found whether you run from the project folder or invoke `vte-ui` from anywhere else (a fresh shell, a shortcut, another terminal tab). If neither location has it, the error message prints both full paths it checked.
