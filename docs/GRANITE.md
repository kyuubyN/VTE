# Multi-architecture support: adding Granite 4.1 3B

Granite 4.1 3B (Q8_0) was added as a deliberate stress test of the codebase's assumptions, all of which had been built and validated against exactly one model. It shares the compiler/IR/executor pipeline with Qwen almost entirely — new code was needed only where the architectures genuinely differ: `granite_mapper.py` (memory plan), `GraniteTokenizer` (vocabulary/chat template), a few extra per-node metadata fields (`rope_type`, `attention_scale`, `residual_scale`) threaded through the existing IR, and a raw-Q8_0 dequant path in the compiler alongside the existing Q4_K/Q6_K one. `SUPPORTED_ARCHITECTURES` (in `sanitizer.py`) and `VTEModel.MODEL_REGISTRY` (in `model.py`) are the two tables that make a new architecture selectable at all — both list Qwen2.5 and Granite side by side, and `model_name=` at `from_pretrained()`/CLI/UI-dropdown time is what picks between them (see [Getting started](USAGE.md)).

## The two headline bugs

Two of the bugs found while bringing Granite up were serious enough to cover in detail here rather than as one-line entries in [Bugs found during development](BUGS.md), because both are the kind of "plausible but wrong" numerical bug that document keeps coming back to — the model ran, produced fluent-looking text, and was still wrong:

- **RoPE rotation convention mismatch (NEOX vs. NORM) — the actual root cause of Granite's incoherent, mixed-language output.** VTE's shared RoPE-applying kernels (`split_k_qkv_pass2`, `fused_norm_matmul_rope`, `rope.hip.template`) only ever implemented one rotation convention: split-half pairing, where dimension `j` of a head pairs with `j + head_dim/2` to form a rotation pair. That's correct for Qwen2.5 — but not universal. Checked against llama.cpp's actual source (`llama_model_rope_type()` in `llama-model.cpp`, not guessed or assumed from a paper) confirmed `LLM_ARCH_QWEN2` maps to `LLAMA_ROPE_TYPE_NEOX` (split-half) while `LLM_ARCH_GRANITE` maps to `LLAMA_ROPE_TYPE_NORM` (interleaved pairing: dimension `2i` pairs with `2i+1`) — two genuinely different conventions, not a parameterization of the same one. Running Granite through the NEOX-only kernels rotated every embedding by the wrong angle at every position beyond the first, and the result looked like a plausible language-model bug (garbled, mixed-language text) rather than an obviously-broken one — exactly the failure mode that makes bugs like this expensive to find without a reference to diff against. Fixed by threading a `rope_type` parameter (0=NEOX, 1=NORM, read from the same per-architecture metadata table as everything else) through all three kernel templates and their Python argument builders, branching to the correct pairing at kernel level. Granite went from garbled output to fully coherent, correct answers immediately after this one fix.
- **`residual_scale` applied to every projection instead of only the ones feeding a residual add.** Granite defines a `residual_scale=0.22` (a value Qwen doesn't have at all) meant to scale a sublayer's output immediately before it's added back into the residual stream — confirmed against `granite.cpp`'s own source (`ggml_scale(cur, f_residual_scale)` called right before `ggml_add(cur, inpSA)` / `ggml_add(cur, ffn_inp)`, and nowhere else). VTE's kernel-argument builder instead applied it unconditionally to every MATMUL node's output, including `gate_proj`/`up_proj`/`q_proj`/`k_proj`/`v_proj` — tensors with no residual add downstream at all. Fixed by gating the real scale value behind the existing residual-fusion lookup (`RESIDUAL_FUSION.get(node.name)`, already used elsewhere to decide which nodes get epilogue-fused with their residual add), falling back to `1.0` for every other node. First-token confidence on a held-out prompt went from ~11% to ~57% after the fix — a large, measurable jump from what looked like a small scoping mistake.

## Two smaller findings

Worth a line each so they aren't silently rediscovered:

- **Raw Q8_0 dequantization only pays off for some tensor shapes, not all of them — measured, not assumed.** Q8_0's block layout (34 bytes / 32 elements) isn't a multiple of 16 bytes, so `gemv_q8_0.hip.template` can't use the same `uint4`-vectorized block reads `gemv_coalesced`/`gemv_q4k` rely on; its per-block unpack loop is scalar. Routing `attn_output` (a smaller GEMV shape) through raw Q8_0 as an experiment measured *slower* than dequantizing to FP16 first and using the existing `gemv_coalesced` kernel (~32.4–36 tok/s vs. a ~36.3+ tok/s FP16 baseline) — the scalar unpack overhead outweighs the bandwidth saved at that shape. `granite_mapper.py::is_raw_q8_0_weight()` excludes all four `attn_*` tensors (`attn_q`, `attn_k`, `attn_v`, `attn_output`) from raw-Q8_0 routing for this reason; only the larger `ffn_gate`/`ffn_up`/`ffn_down`/`token_embd` tensors stay raw, where the bandwidth savings do win.
- **`FallbackExecutor.num_layers` was hardcoded to `28`** (Qwen's actual layer count) instead of being read from model metadata — found incidentally while profiling Granite in eager mode (the profiler's launch counts were suspiciously stuck at a 28-layer-consistent pattern for a model that has 40 layers). Low real-world impact, since the default HIP Graph path was unaffected and `FallbackExecutor` is only the safety net used if graph capture fails — but a genuine silent-truncation bug in that fallback path nonetheless. Fixed to `self.metadata.get('block_count', 28)`.

## Getting Granite's throughput up

Once correctness was established, Granite got the same "measure, don't guess" throughput treatment as Qwen (see [Performance](PERFORMANCE.md)): vectorizing the Split-K QKV Pass1 kernel's inner dot-product loop with `__half2` (pairing two `__half` reads/multiplies per iteration instead of one — safe here because `chunk_size = hidden_size/32` is even for both models, 48 for Qwen and 80 for Granite) delivered a measured ~10% steady-state improvement (~42–44 → ~45–48 tok/s). Granite fits in ~4.8GB VRAM including the SlabAllocator's margin (measured via `hipMemGetInfo`, the real system-wide GPU memory counter, not just VTE's own internally-tracked sub-allocation sum — see [Bugs found during development](BUGS.md) for why those two numbers differ) and reaches the same ~89% of llama.cpp/Ollama's throughput that Qwen does (see the [benchmark](PERFORMANCE.md#benchmark-vte-vs-ollama-llamacpp)) — real evidence that the RoPE/residual-scale fixes above generalize the architecture rather than papering over one model's specific quirks.

## The Flet UI's model-switch race conditions

Letting the desktop UI switch models at runtime (not just at startup) surfaced two related concurrency bugs in `Orchestrator`/`VTEApp` — both real, both reported by direct testing rather than by inspection, and both fixed with the same underlying idea: a monotonically increasing "generation" counter that tags which motor subprocess a message actually came from.

- **A false-positive "motor pipe closed unexpectedly" error on every single model switch**, not an occasional race — this was reported explicitly ("isso sempre acontece") after an initial hypothesis that it was a rare startup-timing race turned out to be wrong. Root cause: `Orchestrator._listener_generation` was incremented *after* `self.motor_process.join(timeout=...)` returned for the old process, but the old `_listen_pipe` background thread's `EOFError` (from that same old pipe legitimately closing during the join wait) could fire while the generation counter still matched — so a perfectly normal shutdown got reported as an unexpected error. Fixed by moving the increment to the very top of `start_motor()`, before the old process is touched at all.
- **Stale messages from an already-replaced motor generation leaking into the new model's session** — not just the error above, but any message, including a legitimate `MotorMsgDone` from the old process arriving after the new one had already started loading. Symptom, reported directly: sending a message to Granite while it was still loading (nothing prevented it) could re-enable the input field via a straggling old-generation "done" message before the new model was actually ready, and in one case produced an unhandled error just from pressing Enter. Fixed two ways together: `_listen_pipe` now re-checks `generation == self._listener_generation` immediately before forwarding *every* message (not just the final error), and a new `model_ready` boolean on `VTEApp` — set `False` on every `reset_for_new_motor()`, `True` only when a current-generation `MotorMsgReady` arrives — independently gates whether `_handle_submit` will send anything at all. `pipe_conn.send()` calls in the UI's event handlers were also wrapped in try/except (`BrokenPipeError`/`OSError`/`EOFError`), since a send racing a subprocess restart can legitimately fail even with both fixes above in place.
- **A visually-alarming crash banner during this same testing turned out not to be a real GPU crash.** A screenshot showed what looked like a WDDM TDR event ("Núcleo da GPU abortado"). Investigated against ground truth rather than trusting the banner's own text: Windows Event Log's System channel (Event ID 4101, the actual TDR event) and Application channel (Event ID 1000, Windows Error Reporting crash events) via `Get-WinEvent -FilterHashtable`, plus checking for a fresh `KernelWatchdog` timeout file. None of the three fired. The banner was a downstream symptom of the same generation-race bug above (a motor process being `.terminate()`d mid-boot during a rapid model switch, surfaced by the UI as an alarming-looking generic error), not a real driver/GPU fault — confirmed once the race itself was fixed and the same test sequence stopped producing the banner at all.
- **A hidden `max_tokens=512` default silently capped every generation, masking two symptoms as one bug report.** `UIMsgPrompt.max_tokens` defaults to `512` in `ipc.py`, and `_handle_submit` never overrode it with the UI's own context-length dropdown value — so long generations got cut off mid-word once they crossed 512 tokens (reported directly, with the actual cut-off text pasted in: a story ending mid-word, "...contra inv"), and separately, the context-length dropdown *looked* broken because changing it had no visible effect on anything (`current_seq_len >= self._context_length` in the decode loop was already correct and unrelated — the cap was hit long before that ceiling could matter). Fixed by sending `max_tokens=self.current_context_length` instead of relying on the dataclass default. Confirmed with a real generation at `context_length=1024`: it now completes naturally (ends in a period, sends `MotorMsgDone(cancelled=False)`) instead of being cut mid-word at 512.
- **Light theme left chat bubbles on their dark-theme colors.** `_make_bubble()` baked the current palette's colors into each bubble at creation time; the theme-toggle handler (`_apply_palette()`) only re-themed the UI chrome, never revisited bubbles already on screen. Fixed by tracking every created bubble (`self.chat_bubbles`) and re-theming each one's background, border, and text colors inside `_apply_palette()`.
- **A generic "loading" screen was reused for model switches, which could be mistaken for the UI having frozen** rather than genuinely loading a different (larger) model — flagged directly as a UX concern, not a bug. `reset_for_new_motor()` now distinguishes initial boot, same-model restart, and model-switch (via a `model_name != self.current_model_name` check made *before* overwriting the tracked name) and shows a distinct title/status message for each.
- **Interface language toggle (pt/en flag button) needed real flag images, not emoji.** Regional-indicator flag emoji (🇺🇸/🇧🇷) don't render reliably in Flutter/Flet on Windows desktop — no bundled color-emoji font ligates those code points into a flag glyph, so the button showed blank. Fixed with two small local SVG assets (`assets/flag_us.svg`, `assets/flag_br.svg`) rendered via `ft.Image`, swapped on click — same approach used for the app's own logo, just applied to the language button.
