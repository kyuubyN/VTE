# Multi-architecture support: adding Qwen 3.5 2B (hybrid Gated DeltaNet)

Qwen 3.5 2B (Q6_K) is a different kind of addition than Granite was. Granite reused the same building blocks (GEMV, RoPE, RMSNorm, softmax attention) with different conventions and dimensions: a new architecture *config*, not new *math*. Qwen 3.5 is a **hybrid**: 24 layers following a fixed pattern, 18 of them `linear_attention` (Gated DeltaNet: a recurrent linear-attention variant with a persistent per-layer state, not the sliding-window KV cache every other layer in this project uses) and 6 `full_attention` (ordinary softmax attention, but with an extra QK-Norm + sigmoid-gate wrinkle Qwen2.5/Granite don't have). Bringing it up meant writing an entire new kernel family: causal depthwise conv1d, the Gated DeltaNet recurrence itself, a gated RMSNorm, none of which existed anywhere else in the codebase before. New code is isolated the same way Granite's was: `qwen3_5_mapper.py` (memory plan + state allocation), `qwen3_5_compute.py` (per-layer graph with the `linear_attention` vs. `full_attention` branch), and five new `.hip.template` kernels, wired into the two existing dispatch tables (`kernel_arg_builder.py::KERNEL_SIGNATURES`, `hip_graph_executor.py::op_to_template`) plus a **third one that turned out to exist** (see below).

## The headline bug: an FP16 buffer read as `float*`, the actual root cause of "the model can't do 'oi'"

The most damaging bug in this port had nothing to do with the novel recurrence math, RoPE, or attention: it was a plain C parameter type mismatch. `gated_delta_recurrent.hip.template` declared its four small per-head parameters (`a`, `b`, `A_log`, `dt_bias`: the decay gate's raw inputs) as `const float* __restrict__`, but every one of those four buffers is actually stored as **FP16** (`__half`, 2 bytes) in VRAM: `a_proj.output`/`b_proj.output` are ordinary MatMul outputs (always FP16 in this project), and `ssm_a`/`ssm_dt.bias` are F32 GGUF weights that get converted to FP16 during loading like any other small weight not routed raw. The kernel was silently reinterpreting pairs of adjacent FP16 values as single garbage float32 numbers: reading `dt_bias` values like `658499.0` and `394786.25` instead of the real (GGUF-confirmed) small values like `3.7` and `-4.97`. This produced a decay gate `g_t = exp(-exp(A_log) * softplus(a + dt_bias))` that was pathologically bimodal (exactly `0.0` or `1.0` for entire heads, never a gradual value) from the very first Gated DeltaNet layer, first position processed.

The symptom was deceptive: the model *ran*, produced fluent-looking BPE tokens, mostly followed a Qwen chat template shape, but degenerated into genuinely incoherent text (`"...Theoremiumster test withy results current...止止...而是字字有差文本结束"`) on prompts as short as "oi", and the same GGUF file run through Ollama on the identical prompt produced a perfectly coherent reply. That side-by-side (same weights, same quantization, different engine) was the single most useful piece of evidence in the whole investigation: it ruled out "this is just what Q6_K quantization sounds like" as an explanation before any kernel was touched.

Finding it took a layer-by-layer, position-by-position teacher-forcing comparison against the real reference (feeding the reference model the *exact* token sequence VTE's own greedy decode produced, single forward pass, `output_hidden_states=True`), and even that only narrowed it to "somewhere in the Gated DeltaNet recurrence" until the actual numeric values of `g_t`/`beta_t` were dumped and compared directly against hand-computed NumPy values from the real GGUF weights. Fixed by changing the four parameters to `const __half* __restrict__` and adding `__half2float()` at each of the four use sites (softplus, decay, beta). Impact, measured the same layer/position way: cosine similarity against the reference went from **0.44–0.98** (worst at "assistant"/`<|im_end|>` transition positions) to **0.85–1.00 everywhere**, and real generation flipped from garbage to genuinely correct: `"A capital da França é"` → `" Paris."`; "oi" → coherent English, no more repeating symbol soup.

## Five more real bugs found bringing the hybrid architecture up

All found and fixed before the FP16 bug above, each with real data confirming both the break and the fix: none of these were hypothetical:

- **QKV fusion firing on `linear_attention` layers by name alone.** The "deep fusion" optimization (RMSNorm+QKV+RoPE in one launch, built for Qwen2.5/Granite) intercepted any node literally named `attn_norm`, without checking whether that layer actually has separate Q/K/V weights to fuse. Gated DeltaNet layers also have a node called `attn_norm`, followed by a completely different shape (`attn_qkv.weight`, one fused matrix, not three), so the interception zeroed the real RMSNorm output for those layers and propagated zero through the rest of the model. This was the root cause of the very first, most catastrophic test result (`"!!!!!!!!"` repeated). Fixed with a guard requiring `f"blk.{layer_idx}.attn_q.weight"` to actually exist in the tensor mapping (and `rope_type != 2`) before attempting the fusion, in both `fallback_executor.py` and `hip_graph_executor.py`.
- **A third, previously-undiscovered kernel dispatch table.** Two tables were already known to need updating for any new `NodeType` (`kernel_arg_builder.py::KERNEL_SIGNATURES` and `hip_graph_executor.py::op_to_template`), but `FallbackExecutor._get_or_compile_kernel` turned out to keep its *own*, independent `template_map` dict, never touched by either of the other two updates. `CAUSAL_CONV1D`/`LINEAR_ATTENTION`/`RMSNORM_GATED` silently no-op'd in the eager execution path (only a debug-level log line, no error) until this third copy was found and updated too.
- **A real buffer overflow from using the wrong raw-weight predicate at load time.** `weight_loader.py` called the Qwen2.5-specific `is_raw_q6k_weight()` (restricted to `ffn_down`/`token_embd` because Qwen2.5's GGUF mixes Q4_K and Q6_K) instead of the already-unified raw-weight sets `model.py` had computed for the actual architecture in use. Large Q6_K matrices (`attn_qkv`, `attn_gate`, `ffn_gate`, `ffn_up`) got dequantized to FP16 (~2.4x their raw size) and written into a VRAM slot the mapper had only sized for the *raw* format: a real overflow, corrupting adjacent VRAM. This produced early numeric values roughly 400x the expected scale, and was the first clue that something more structural than "normal FP16/quantization noise" was wrong. Fixed by threading the pre-computed unified sets into `GGUFWeightLoader.__init__` instead of recomputing a narrower one locally.
- **`attn_q`/`attn_k`/`attn_v`/`attn_output.weight` routed raw when the fused kernel couldn't read them.** `qwen3_5_mapper.py`'s own `is_raw_q6k_weight()` was dtype-only, missing the by-name exclusion `qwen_mapper.py`/`granite_mapper.py` already had for these four tensors. Left raw, they got read by the fused QKV+RoPE kernel as plain `__half*` with zero dequant logic: producing NaN from the first `full_attention` layer onward. Fixed with the same by-name exclusion (later made moot for Qwen3.5 specifically, since the fusion is now unconditionally disabled whenever `rope_type == 2`. See below, but kept as defense in depth).
- **An entire missing architecture piece: QK-Norm, a double-width `q_proj`, and a sigmoid gate.** Confirmed line-by-line against the real `Qwen3_5Attention.forward` (and independently against the GGUF's own `attn_q.weight` shape: out_features=4096, not the assumed 2048) that `q_proj` actually produces `query` and `gate` interleaved per-head in one double-width matrix; `query`/`key` get a per-head RMSNorm (`q_norm`/`k_norm`, real weights `attn_q_norm.weight`/`attn_k_norm.weight` present in the GGUF) *before* RoPE; and the attention output gets multiplied by `sigmoid(gate)` before `o_proj`. This wasn't a bug in existing code, it was an entire piece of the graph that had never been built: without it, layer 3 (the first `full_attention` layer) ran without crashing but at roughly 9x the expected activation scale. Implemented as two new kernels (`per_head_rmsnorm.hip.template`, `sigmoid_gate_mul.hip.template`) plus a rewritten `full_attention` branch in `qwen3_5_compute.py`.

## State persistence: a bug that only shows up on the SECOND `generate()` call

Gated DeltaNet's per-layer state (`[16 heads, 128, 128]` FP32, persists across the whole generation: unlike the KV cache, which implicitly restarts at `kv_offset=0` every call) was zeroed via `hipMemset` exactly once, at model load. A single `generate()` call always looked correct, because the state starts zeroed either way: the bug only appears on a *second* call against an already-loaded model instance, which would silently inherit the first generation's leftover state instead of starting clean. Caught by asking the direct question "is the state buffer actually zeroed at the start of every new sequence, or just once?" rather than by a symptom: this is the kind of bug that's easy to ship because the most common manual test (load once, generate once, exit) never exercises it. Fixed with `VTEModel._reset_gdn_state_if_needed()`, called at the top of every `generate()`, gated on `architecture == "qwen35"`. Verified with two consecutive `generate()` calls (same prompt, same loaded instance, no reload in between) producing byte-identical output.

## Streaming/UX bugs found via real Flet UI testing

Four more bugs (2026-07 session), found only once the model was exercised through the actual desktop chat UI with a real chat-formatted prompt and streamed output, rather than the short synthetic sequences used above:

- **`decode()` silently ate the `<think>`/`</think>` tokens along with the ChatML markers, breaking thinking-mode detection.** `Qwen3_5Tokenizer.decode()` treated every entry in `special_tokens` the same way (skip it entirely): correct for `<|im_start|>`/`<|im_end|>`/`<|endoftext|>`, wrong for `<think>`/`</think>`, which `ThinkingSectionScanner` (`vte/core/thinking_scanner.py`) needs to see as literal text in the decoded stream to split "thinking" from "answer" sections (a decision made when the scanner was written, before Qwen3.5 could run for real, that the doc comment never got updated to reflect). With the tags silently dropped, the scanner's `_in_thinking` flag never flipped, so *everything* (including a model's actual internal reasoning draft, which is deliberately less polished than a final answer) got shown to the user as if it were the final response, making the model look far less coherent than it actually was. Fixed by special-casing exactly those two tokens in `decode()`/`decode_bytes()` to survive as literal text, while every other special token is still suppressed as before.
- **The default `repetition_penalty=1.1` (well-calibrated for Qwen2.5/Granite) is too weak for Qwen3.5 at greedy (`temperature=0`) decode on longer generations.** With 1.1, a real prompt reliably degenerated into a repetition loop after a few hundred tokens (`"**Cultura:** **Cultura:**..."` collapsing into just `"**"` repeated dozens of times). Measured 1.1 / 1.3 / 1.5 directly against the same prompt: 1.3 produced a complete, coherent response that ended naturally with no loop; 1.5 over-corrected into punctuation/emoji noise instead of words. Fixed with a per-architecture default (`VTEModel._default_repetition_penalty()`): 1.3 for `qwen35`, 1.1 unchanged for everything else, rather than changing the global default and risking a regression on the two models it was already tuned for.
- **Streamed emoji and accented characters corrupted into `�` mid-response.** Noticed on Qwen3.5 (which uses emoji far more than the other two models), but this one is shared code, not Qwen3.5-specific: same root cause and fix apply to Qwen2.5 and Granite too, see the general write-up in [Bugs found during development](BUGS.md).
- **Fixed (2026-07): the real root cause of Qwen3.5's incoherent-but-fluent output, a wrong buffer read in the Split-KV attention fast path, not FP16 precision drift as long assumed.** Reported directly: a chat-templated prompt (system prompt + `enable_thinking=True`) reliably produced grammatically fluent but semantically wrong answers ("Qual a capital da França?" → confident nonsense about Brazil), with the `<think>` block always coming out empty. `docs/QWEN35.md` (this file) previously attributed this to "diffuse FP16 precision loss accumulated over 24 layers": measured cosine similarity against a saved single-forward-pass reference (`scripts/diagnostics/qwen35_reference/long_gen_vte_prompt_all_layers_ref.npy`, 25×10×2048, embedding + 24 layers) seemed consistent with that story (0.85–1.00 per layer). But teacher-forcing the real 10-token prompt layer-by-layer (`scripts/diagnostics/diag_qwen35_teacherforce_layers.py`) showed the degradation was *not* smooth: layer 7 alone dropped cosine by +0.108 in a single step, and the final layer (23) by +0.081, landing at 0.57 overall: a discrete cliff, not diffuse noise, and both drops localized to `full_attention` layers (3/7/11/15/19/23), never the 18 Gated DeltaNet layers (which, if anything, *recovered* cosine).

  Isolating attention itself (`diag_qwen35_l7_attention_isolada.py`: reconstruct causal softmax attention in pure NumPy from VTE's own live Q/K/V buffers, compare against VTE's actual `attention.output`) found cosine 0.294 against layer 7's own attention output: a catastrophic, structural mismatch, not rounding noise. A KV-cache round-trip check (`diag_qwen35_kvcache_roundtrip.py`, read back what's actually stored in the persistent K/V cache vs. the live per-position values) narrowed it further: V matched bit-exact at every position, K was off by 4–6 (absolute, on values that size) at *every single position*, a systematic addressing bug, not accumulated error.

  Root cause: `SplitKVAttentionDispatcher.build_launches()` (`vte/core/split_kv_attention.py`, the fast attention path used by `HIPGraphExecutor` in production, `use_hip_graph=True`) reconstructed its Q/K/V tensor names by a fixed naming *convention* (`blk.{layer}.q_proj.output`/`k_proj.output`) instead of reading the real names from the compute graph node. That convention is exactly right for Qwen2.5/Granite, where RoPE modifies `q_proj.output`/`k_proj.output` in place (no other step in between). But Qwen3.5's `full_attention` layers have an extra per-head RMSNorm (`q_norm`/`k_norm`) *between* the projection and RoPE: real Q/K used for attention live in `q_norm.output`/`k_norm.output`, separate buffers, with RoPE modifying *those* in place. The fast path was silently attending over stale, un-normalized, un-rotated K (and, for Q, reading the wrong-width buffer entirely: `q_proj.output` is double-width, `[query|gate]` interleaved per head before the per-head norm splits them apart). V happened to match because Qwen3.5's architecture has no analogous per-head norm for V (`v_proj.output` really is the final V, same as Qwen2.5/Granite): which is exactly why the K-only mismatch was the tell. `kernel_arg_builder.py`'s slower fallback attention path was never affected: it already read tensor names from `node.input_tensors`, correctly, the whole time; only the fast Split-KV path used the hardcoded convention.

  Fixed by threading the graph's real tensor names (`node.input_tensors[0..2]`) through `HIPGraphExecutor` into `build_launches()` (new `q_name`/`k_name`/`v_name` parameters, defaulting to the old convention when omitted: zero behavior change for Qwen2.5/Granite, verified by regression). Post-fix, the same layer-7 attention isolation test lands at cosine 1.000000 (max abs diff 0.002, pure FP16 rounding), and the full teacher-forced layer sweep shows a smooth, incremental curve with no discrete drops anywhere (final layer cosine 0.79–0.91 across positions, up from 0.57–0.85). Real generation: "Qual a capital da França?" now answers "**Paris**" with coherent elaboration (previously confident nonsense about Brazil); "Escreva um poema curto sobre o mar" produces genuine on-topic poetry (previously unrelated rambling). Regression-tested clean across all four other models/variants (Qwen2.5 1.5B/7B/0.5B, Granite): this bug was Qwen3.5-specific because it's the only architecture with a norm step between projection and RoPE.

  **What this means for the previous "FP16 precision ceiling" framing above and in the temperature/repetition-penalty write-ups**: that framing is now known to be at least partially wrong, a meaningful fraction (likely the majority) of the "noise" attributed to accumulated FP16 drift was actually this one structural bug, present in every single generation this model ever produced in this project. Real per-layer FP16 precision loss may still exist at some smaller scale (not yet re-measured post-fix), but it is no longer the leading explanation for incoherent output, and the `temperature=0`/`repetition_penalty=1.3` mitigations described below should be re-evaluated against this fix before being treated as permanent.

  **Post-fix re-measurement (2026-07, same session): the residual noise, quantified with a real 1000+ token ground truth.** After the Split-KV fix, a "write a long story" prompt at `temperature=0` still degraded past ~700 tokens: not into the old repetition loop or the old catastrophic incoherence, but into vague philosophical rambling that never engages with the actual prompt, eventually trailing into word-salad and keyboard-mashing noise by the very end. Self-consistency checks (attention-output magnitude, per-head-norm magnitude, Gated DeltaNet state magnitude, final hidden-state magnitude: all tracked every 25 tokens across the full ~975-token generation) showed **no NaN/Inf, no runaway growth, no discrete spike anywhere**: ruling out both an FP16 overflow/underflow signature and a "RoPE base misconfigured" signature (independently confirmed: the real GGUF's `rope.freq_base=1e7`/`rope.dimension_count=64` exactly match what the engine already uses). This meant the divergence had to be checked directionally, not just by magnitude.

  Re-installed `torch`+`transformers` temporarily (same disposable-reference discipline as every other measurement in this document) specifically to get a **real ground truth beyond the original 10-token sample**: fed the exact ~1082 token sequence VTE itself generated (prompt + its own output, teacher-forced) through a single real forward pass (`output_hidden_states=True`, avoiding the known-broken `generate()` path), and compared cosine similarity against VTE's own captured hidden states at *every layer, every position* of that same real generation: not a synthetic short prompt this time, the actual failing sequence.

  Two findings, matching neither of the two "textbook" divergence signatures cleanly:
  - **Per-layer**: no discrete cliff anywhere (confirming the Split-KV fix holds even over long sequences), cosine averages ~0.93–0.96 per layer from L1 through L23, down from a near-perfect 0.9998 at the embedding. This ~0.94 per-layer average, sustained across 24 layers of composition, *is* the real residual "FP16 precision ceiling": genuine, but far smaller than what the old (buggy) measurements suggested.
  - **By position**: not a clean monotonic decline either, the final layer's cosine bounces in a ~0.89–0.95 band across the *entire* 1082-token sequence (lowest segment was actually the *first* 100 tokens at 0.8955, not the last; the last segment lands at 0.9112, only mildly below the middle-of-sequence average). This rules out simple geometric/cumulative drift in the hidden states themselves.

  **Interpretation**: the mechanism is closer to autoregressive exposure bias riding on top of a real-but-modest, roughly constant per-layer FP16 noise floor, not a growing numerical error. Cosine ~0.93–0.96 is high enough that most individual argmax decisions still land correctly, but not high enough to guarantee it every time over hundreds of sequential greedy choices: once one pick diverges from what the reference model would have chosen, every subsequent token is generated from a context the model never saw in training (true exposure bias, not a VTE-specific bug), and greedy decoding has no mechanism to recover once several of those small missteps compound narratively. The hidden states themselves stay numerically bounded and reasonable throughout (matching the magnitude self-consistency check): it's the *direction*, compounding through autoregression, that eventually drifts into content that doesn't track the prompt. This is a real, still-open quality limitation, but it is now precisely characterized rather than attributed to a vague "precision ceiling," and it is a substantially smaller effect than what the pre-fix measurements (contaminated by the Split-KV bug) had suggested.

- **Fixed (2026-07): the "###\n\n###\n\n###..." repetition collapse in very long generations was a sampler design flaw, not a numerical/state saturation issue as first suspected.** Reported directly: a "write a long story" prompt at `context_length=2048` degenerated into an infinite `"### ### ###..."` loop from roughly token ~700 onward, still stuck at the 1939-token budget ceiling. First hypothesis (Gated DeltaNet's persistent recurrent state drifting into a degenerate/saturated regime over hundreds of steps, compounding the FP16 precision noise documented above) was tested directly with `scripts/diagnostics/diag_qwen35_logit_confidence.py`: logged the top1-vs-top2 logit gap at every step of a 1200-token greedy generation with repetition penalty disabled entirely. The gap stayed in a modest, stable 1–6 range throughout, including at and after the point of collapse (gap=1.6 deep into the loop, actually *lower* than gap=7.0 earlier in the same generation), directly refuting "runaway confidence." What the trace showed instead: the model settles into a short, low-margin cycle of ~8 tokens (`"Okay, I need to write. *\n"` repeating), each individually only moderately preferred over the runner-up. `Sampler.sample()`'s repetition penalty (`vte/core/sampler.py`) applied a *flat, one-time* discount per unique token regardless of how many times it had already repeated (`set(generated_tokens)`, then a single fixed `/repetition_penalty`): against a persistent multi-token cycle, this discount landed once and never escalated, so the same ~8 tokens kept winning against each other indefinitely. Fixed by making the penalty scale with actual repeat count within a recent sliding window (`REPETITION_WINDOW=512` tokens, so a legitimately common word used many times early in a long story isn't penalized once it scrolls out of the window): `effective_penalty = repetition_penalty ** min(count, REPETITION_COUNT_CAP)` (`REPETITION_COUNT_CAP=10`, preventing an absurd exponent for genuinely frequent common words). At `count=1` (a token seen once) the exponent is 1 and the result is identical to the old flat discount: no behavior change for the common case. Verified: the same story prompt that previously ran to the full 1939-token budget stuck in the `"###"` loop now stops naturally at 352 tokens (hits a real stop token) with no repeated-token cycle. Regression-tested across all four registered models (Qwen2.5 1.5B/0.5B, Granite, Qwen3.5) with short prompts: no change in output. **Not fully resolved**: the story's actual content is still poor (rambles about abstract themes instead of following the prompt), a real but *different*, softer quality problem, not the hard repetition-loop bug this fix targeted.
- **Any `temperature > 0` reliably degenerates Qwen3.5's output: a real precision ceiling, not a parameter to tune away.** The Flet UI's motor calls `generate()` without an explicit `temperature`, so it inherited the global default (`0.7`, ordinary stochastic sampling). Reported directly: "oi" produced a long, fluent-looking but semantically empty ramble that drifted through French, misidentified the model's own name, and never actually answered the greeting. Swapping `repetition_penalty` between 1.0/1.1/1.3 at `temperature=0.7` changed the *flavor* of the degeneration (word loops vs. language-switching vs. unpunctuated run-on sentences) but never fixed it: a strong signal the problem wasn't repetition penalty at all. Swept `temperature` directly (0.1, 0.2, 0.3, 0.5, 0.7) with the same prompt: every value `>0` degenerated in some way; only `temperature=0.0` (greedy) was consistently coherent, matching the "gradual, not sudden" cosine-similarity noise (0.85–1.00, not 0.99–1.00) already documented in the FP16 recurrence fix above: small enough to never flip which token *wins* argmax, large enough to visibly distort the probability tail that stochastic sampling draws its variety from. Mitigated (not fixed) the same way as the `repetition_penalty` default: `VTEModel._default_temperature()` returns `0.0` for `qwen35`, `0.7` unchanged for Qwen2.5/Granite. This is a real, accepted trade-off, not a resolved bug: Qwen3.5 responses are now deterministic (same prompt always produces the same answer) until the underlying per-layer FP16 precision loss is reduced enough that sampling above `temperature=0` stops being visibly risky.

## "Talks nonsense with a large context": it wasn't the context, it was the response budget

Reported (2026-07): selecting a large `context_length` in the UI dropdown made even "oi" degenerate into incoherent text. Isolated with `scripts/diagnostics/diag_qwen35_context_bug.py` (same prompt, same seed, `context_length` from 2048 to 16384, short and medium generations): **bit-identical output across the entire range**, refuting the hypothesis that the KV cache/RoPE cache size itself caused the degeneration.

Real cause, confirmed with `scripts/diagnostics/diag_qwen35_long_gen.py`: the UI (`vte/ui/app.py`) sent `max_tokens=current_context_length`, meaning picking `context_length=8192` made **any** message, including "oi", request a generation of up to 8192 tokens. Running that generation to completion: coherent up to ~300 tokens, degrading into generic filler by ~500-1000, and from ~1500-2000 tokens on entering the repetition loop already documented above (`"...enhancing enhancing enhancing..."` continuing for thousands of tokens), the same greedy-decode-without-natural-EOS loop, not a new kernel bug. The user only saw it because the UI let the generation run long enough to get there, even on a greeting.

Fixed in `vte/ui/app.py` with `MAX_SINGLE_RESPONSE_TOKENS = 2048`: the per-response token ceiling is now `min(context_length, MAX_SINGLE_RESPONSE_TOKENS)`, decoupling "KV cache capacity" from "single-response token budget". For the dropdown values ≤2048 (512/1024/2048) behavior is identical to before; only 4096/8192 get a reduced ceiling.

Bonus, defense in depth: `VTEModel.generate()` had NO check that the tokenized prompt fit within `context_length` before prefill, a prompt larger than the allocated KV cache/RoPE cache would write `kv_offset` past the buffer, silently corrupting the Activation Arena (the decode loop had protected itself against this from early on; only prefill didn't). Added a fail-fast `HIPSafetyError` at the top of `generate()` when `len(prompt_tokens) >= context_length`, the same clear-error discipline already used in the mappers' preventive OOM check.

## Measured before implementing: FP32 residual stream would have limited effect

A natural next idea after quantifying the residual noise above: keep the residual stream (the tensor that accumulates via the `ADD` after each attention/FFN sub-layer, flowing through all 24 layers) in FP32 in VRAM instead of truncating to FP16 at every layer boundary, on the theory that repeated FP16 rounding of that specific tensor is what compounds into the ~0.94 average cosine gap. Measured this **before** committing to the implementation (real GEMV/RMSNorm/attention re-plumbing across every layer, non-trivial and not risk-free): took the real FP32 hidden states already captured from `transformers` (`hf_long_hidden_states.npy`) and computed the cosine similarity between each value and its own FP16-round-tripped version (`x.astype(np.float16).astype(np.float32)`), i.e. the absolute floor of noise a *single* FP16 storage round-trip can introduce, with zero other error sources.

Result: that floor is **cosine 1.000000** at every single layer: FP16 has more than enough precision to store one activation value with negligible relative error; the noise from pure storage rounding is not measurable at this scale. The real VTE-vs-reference gap, by contrast, is already down to **cosine 0.986 at Layer 0** and settles around 0.93–0.96 for the rest of the stack: two to three orders of magnitude bigger than what residual-stream rounding alone could ever produce, and already present after just one layer (not something that visibly compounds layer-over-layer at a rate consistent with "one more truncation per layer").

**Conclusion**: the residual noise is not primarily a storage-precision artifact of truncating the residual stream to FP16: it is dominated by something that introduces real value-level error into what a layer computes in the first place, most plausibly the lossy weight quantization itself (Q6_K/Q4_K/Q8_0 all approximate the true weight values, not just their storage format) combined with FP16 GEMV accumulation noise across many kernels per layer. An FP32 residual stream would not address that. **Not implemented**: the measurement showed it wasn't worth the real engineering cost and regression risk (touching every layer's tensor plumbing) for a fix that targets the wrong variable. Reducing this further for real would mean running weights at higher precision (Q8_0 or FP16 instead of Q6_K): a genuine VRAM/speed trade-off, not a kernel correctness fix.

## Known limitation: only the Bartowski Q6_K conversion is validated to work

While investigating the residual FP16 noise (below), a second GGUF of the same nominal model (Qwen3.5 2B Q4_K_M from Unsloth, as opposed to the Q6_K from Bartowski this project has always used) was tested as a cheap alternative before committing to a bigger engine change. Two real, separate findings came out of that detour, both worth recording even though the conversion itself wasn't adopted:

- **GGUF conversions of the same nominal model are not guaranteed structurally identical.** The first Unsloth file downloaded had **25** transformer blocks (`blk.0`..`blk.24`) instead of 24, plus tensors in `Q5_0` that this project had never needed to read before (fixed in passing. See the Q5_0 dequantizer entry in [Bugs found during development](BUGS.md)). A second Unsloth download (Q4_K_M, still labeled "Qwen3.5 2B", `general.architecture=qwen35`) did match the expected 24-block layout, but used `Q5_K` for `attn_qkv`/`ssm_out` on the Gated DeltaNet layers: a quantization type this project also hadn't needed before. `dequantize_q5_k` was implemented and validated with **zero** difference against `gguf`'s own reference dequantization on a real tensor from that exact file: the math is confirmed correct in isolation.
- **Despite that, loading the Unsloth Q4_K_M file end-to-end produced immediate, total incoherence**: not the gradual philosophical-rambling-then-noise pattern documented above, but garbage output (mixed-script noise) from the very first generated token, on a simple factual prompt that the Bartowski Q6_K file answers correctly. Tokenizer round-trip and all GGUF metadata (head counts, RoPE theta/dimension, `full_attention_interval`) checked out identical to the working file, and the Q5_K dequant validated correct in isolation, but the full model still failed completely. The root cause was not isolated before the investigation was stopped; it could be a VTE bug not yet triggered by any previously-tested GGUF, or a genuine incompatibility in that specific community conversion for this still-fairly-new hybrid architecture (Gated DeltaNet quantization/export tooling is less mature than for standard transformers).

**Practical guidance**: only `Qwen3.5-2B-Q6_K.gguf` (Bartowski) is validated to work correctly with this engine. Swapping in a different GGUF of "the same model" from a different converter/quantization is not a safe drop-in: verify block count, dtype mix, and actual generation quality before trusting it. The Q5_K dequantizer added during this investigation is kept (correct, validated, harmless, and may be needed again), but no Q4_K_M variant of Qwen3.5 is currently registered in `MODEL_REGISTRY`: this is a deliberate, temporary scope decision, not an oversight, pending a dedicated investigation into why that specific conversion fails outright.
