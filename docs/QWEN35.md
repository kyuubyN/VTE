# Multi-architecture support: adding Qwen 3.5 2B (hybrid Gated DeltaNet)

Qwen 3.5 2B (Q6_K) is a different kind of addition than Granite was. Granite reused the same building blocks (GEMV, RoPE, RMSNorm, softmax attention) with different conventions and dimensions — a new architecture *config*, not new *math*. Qwen 3.5 is a **hybrid**: 24 layers following a fixed pattern, 18 of them `linear_attention` (Gated DeltaNet — a recurrent linear-attention variant with a persistent per-layer state, not the sliding-window KV cache every other layer in this project uses) and 6 `full_attention` (ordinary softmax attention, but with an extra QK-Norm + sigmoid-gate wrinkle Qwen2.5/Granite don't have). Bringing it up meant writing an entire new kernel family — causal depthwise conv1d, the Gated DeltaNet recurrence itself, a gated RMSNorm — that nothing else in the codebase needed before. New code is isolated the same way Granite's was: `qwen3_5_mapper.py` (memory plan + state allocation), `qwen3_5_compute.py` (per-layer graph with the `linear_attention` vs. `full_attention` branch), and five new `.hip.template` kernels, wired into the two existing dispatch tables (`kernel_arg_builder.py::KERNEL_SIGNATURES`, `hip_graph_executor.py::op_to_template`) plus a **third one that turned out to exist** (see below).

Reference implementation used throughout: `transformers`' real `modeling_qwen3_5.py` source, installed **temporarily** (CPU-only, explicit re-confirmation before every reinstall, fully uninstalled after each use) — the same disposable-reference discipline already used for Ollama comparisons elsewhere in this project. Every numeric claim below was checked against that real reference, not derived from documentation or guesswork.

## The headline bug: an FP16 buffer read as `float*` — the actual root cause of "the model can't do 'oi'"

The most damaging bug in this port had nothing to do with the novel recurrence math, RoPE, or attention — it was a plain C parameter type mismatch. `gated_delta_recurrent.hip.template` declared its four small per-head parameters (`a`, `b`, `A_log`, `dt_bias` — the decay gate's raw inputs) as `const float* __restrict__`, but every one of those four buffers is actually stored as **FP16** (`__half`, 2 bytes) in VRAM: `a_proj.output`/`b_proj.output` are ordinary MatMul outputs (always FP16 in this project), and `ssm_a`/`ssm_dt.bias` are F32 GGUF weights that get converted to FP16 during loading like any other small weight not routed raw. The kernel was silently reinterpreting pairs of adjacent FP16 values as single garbage float32 numbers — reading `dt_bias` values like `658499.0` and `394786.25` instead of the real (GGUF-confirmed) small values like `3.7` and `-4.97`. This produced a decay gate `g_t = exp(-exp(A_log) * softplus(a + dt_bias))` that was pathologically bimodal (exactly `0.0` or `1.0` for entire heads, never a gradual value) from the very first Gated DeltaNet layer, first position processed.

The symptom was deceptive: the model *ran*, produced fluent-looking BPE tokens, mostly followed a Qwen chat template shape — but degenerated into genuinely incoherent text (`"...Theoremiumster test withy results current...止止...而是字字有差文本结束"`) on prompts as short as "oi", and the same GGUF file run through Ollama on the identical prompt produced a perfectly coherent reply. That side-by-side (same weights, same quantization, different engine) was the single most useful piece of evidence in the whole investigation — it ruled out "this is just what Q6_K quantization sounds like" as an explanation before any kernel was touched.

Finding it took a layer-by-layer, position-by-position teacher-forcing comparison against the real reference (feeding the reference model the *exact* token sequence VTE's own greedy decode produced, single forward pass, `output_hidden_states=True`) — and even that only narrowed it to "somewhere in the Gated DeltaNet recurrence" until the actual numeric values of `g_t`/`beta_t` were dumped and compared directly against hand-computed NumPy values from the real GGUF weights. Fixed by changing the four parameters to `const __half* __restrict__` and adding `__half2float()` at each of the four use sites (softplus, decay, beta). Impact, measured the same layer/position way: cosine similarity against the reference went from **0.44–0.98** (worst at "assistant"/`<|im_end|>` transition positions) to **0.85–1.00 everywhere**, and real generation flipped from garbage to genuinely correct: `"A capital da França é"` → `" Paris."`; "oi" → coherent English, no more repeating symbol soup.

## Five more real bugs found bringing the hybrid architecture up

All found and fixed before the FP16 bug above, each with real data confirming both the break and the fix — none of these were hypothetical:

- **QKV fusion firing on `linear_attention` layers by name alone.** The "deep fusion" optimization (RMSNorm+QKV+RoPE in one launch, built for Qwen2.5/Granite) intercepted any node literally named `attn_norm`, without checking whether that layer actually has separate Q/K/V weights to fuse. Gated DeltaNet layers also have a node called `attn_norm` — followed by a completely different shape (`attn_qkv.weight`, one fused matrix, not three) — so the interception zeroed the real RMSNorm output for those layers and propagated zero through the rest of the model. This was the root cause of the very first, most catastrophic test result (`"!!!!!!!!"` repeated). Fixed with a guard requiring `f"blk.{layer_idx}.attn_q.weight"` to actually exist in the tensor mapping (and `rope_type != 2`) before attempting the fusion, in both `fallback_executor.py` and `hip_graph_executor.py`.
- **A third, previously-undiscovered kernel dispatch table.** Two tables were already known to need updating for any new `NodeType` (`kernel_arg_builder.py::KERNEL_SIGNATURES` and `hip_graph_executor.py::op_to_template`) — but `FallbackExecutor._get_or_compile_kernel` turned out to keep its *own*, independent `template_map` dict, never touched by either of the other two updates. `CAUSAL_CONV1D`/`LINEAR_ATTENTION`/`RMSNORM_GATED` silently no-op'd in the eager execution path (only a debug-level log line, no error) until this third copy was found and updated too.
- **A real buffer overflow from using the wrong raw-weight predicate at load time.** `weight_loader.py` called the Qwen2.5-specific `is_raw_q6k_weight()` (restricted to `ffn_down`/`token_embd` because Qwen2.5's GGUF mixes Q4_K and Q6_K) instead of the already-unified raw-weight sets `model.py` had computed for the actual architecture in use. Large Q6_K matrices (`attn_qkv`, `attn_gate`, `ffn_gate`, `ffn_up`) got dequantized to FP16 (~2.4x their raw size) and written into a VRAM slot the mapper had only sized for the *raw* format — a real overflow, corrupting adjacent VRAM. This produced early numeric values roughly 400x the expected scale, and was the first clue that something more structural than "normal FP16/quantization noise" was wrong. Fixed by threading the pre-computed unified sets into `GGUFWeightLoader.__init__` instead of recomputing a narrower one locally.
- **`attn_q`/`attn_k`/`attn_v`/`attn_output.weight` routed raw when the fused kernel couldn't read them.** `qwen3_5_mapper.py`'s own `is_raw_q6k_weight()` was dtype-only, missing the by-name exclusion `qwen_mapper.py`/`granite_mapper.py` already had for these four tensors. Left raw, they got read by the fused QKV+RoPE kernel as plain `__half*` with zero dequant logic — producing NaN from the first `full_attention` layer onward. Fixed with the same by-name exclusion (later made moot for Qwen3.5 specifically, since the fusion is now unconditionally disabled whenever `rope_type == 2` — see below — but kept as defense in depth).
- **An entire missing architecture piece: QK-Norm, a double-width `q_proj`, and a sigmoid gate.** Confirmed line-by-line against the real `Qwen3_5Attention.forward` (and independently against the GGUF's own `attn_q.weight` shape: out_features=4096, not the assumed 2048) that `q_proj` actually produces `query` and `gate` interleaved per-head in one double-width matrix; `query`/`key` get a per-head RMSNorm (`q_norm`/`k_norm`, real weights `attn_q_norm.weight`/`attn_k_norm.weight` present in the GGUF) *before* RoPE; and the attention output gets multiplied by `sigmoid(gate)` before `o_proj`. This wasn't a bug in existing code, it was an entire piece of the graph that had never been built — without it, layer 3 (the first `full_attention` layer) ran without crashing but at roughly 9x the expected activation scale. Implemented as two new kernels (`per_head_rmsnorm.hip.template`, `sigmoid_gate_mul.hip.template`) plus a rewritten `full_attention` branch in `qwen3_5_compute.py`.

## State persistence: a bug that only shows up on the SECOND `generate()` call

Gated DeltaNet's per-layer state (`[16 heads, 128, 128]` FP32, persists across the whole generation — unlike the KV cache, which implicitly restarts at `kv_offset=0` every call) was zeroed via `hipMemset` exactly once, at model load. A single `generate()` call always looked correct, because the state starts zeroed either way — the bug only appears on a *second* call against an already-loaded model instance, which would silently inherit the first generation's leftover state instead of starting clean. Caught by asking the direct question "is the state buffer actually zeroed at the start of every new sequence, or just once?" rather than by a symptom — this is the kind of bug that's easy to ship because the most common manual test (load once, generate once, exit) never exercises it. Fixed with `VTEModel._reset_gdn_state_if_needed()`, called at the top of every `generate()`, gated on `architecture == "qwen35"`. Verified with two consecutive `generate()` calls (same prompt, same loaded instance, no reload in between) producing byte-identical output.

## Streaming/UX bugs found via real Flet UI testing

Four more bugs (2026-07 session), found only once the model was exercised through the actual desktop chat UI with a real chat-formatted prompt and streamed output, rather than the short synthetic sequences used above:

- **`decode()` silently ate the `<think>`/`</think>` tokens along with the ChatML markers, breaking thinking-mode detection.** `Qwen3_5Tokenizer.decode()` treated every entry in `special_tokens` the same way (skip it entirely) — correct for `<|im_start|>`/`<|im_end|>`/`<|endoftext|>`, wrong for `<think>`/`</think>`, which `ThinkingSectionScanner` (`vte/core/thinking_scanner.py`) needs to see as literal text in the decoded stream to split "thinking" from "answer" sections (a decision made when the scanner was written, before Qwen3.5 could run for real, that the doc comment never got updated to reflect). With the tags silently dropped, the scanner's `_in_thinking` flag never flipped, so *everything* — including a model's actual internal reasoning draft, which is deliberately less polished than a final answer — got shown to the user as if it were the final response, making the model look far less coherent than it actually was. Fixed by special-casing exactly those two tokens in `decode()`/`decode_bytes()` to survive as literal text, while every other special token is still suppressed as before.
- **The default `repetition_penalty=1.1` (well-calibrated for Qwen2.5/Granite) is too weak for Qwen3.5 at greedy (`temperature=0`) decode on longer generations.** With 1.1, a real prompt reliably degenerated into a repetition loop after a few hundred tokens (`"**Cultura:** **Cultura:**..."` collapsing into just `"**"` repeated dozens of times). Measured 1.1 / 1.3 / 1.5 directly against the same prompt: 1.3 produced a complete, coherent response that ended naturally with no loop; 1.5 over-corrected into punctuation/emoji noise instead of words. Fixed with a per-architecture default (`VTEModel._default_repetition_penalty()`) — 1.3 for `qwen35`, 1.1 unchanged for everything else — rather than changing the global default and risking a regression on the two models it was already tuned for.
- **Streamed emoji and accented characters corrupted into `�` mid-response.** Noticed on Qwen3.5 (which uses emoji far more than the other two models), but this one is shared code, not Qwen3.5-specific — same root cause and fix apply to Qwen2.5 and Granite too, see the general write-up in [Bugs found during development](BUGS.md).
- **Any `temperature > 0` reliably degenerates Qwen3.5's output — a real precision ceiling, not a parameter to tune away.** The Flet UI's motor calls `generate()` without an explicit `temperature`, so it inherited the global default (`0.7`, ordinary stochastic sampling). Reported directly: "oi" produced a long, fluent-looking but semantically empty ramble that drifted through French, misidentified the model's own name, and never actually answered the greeting. Swapping `repetition_penalty` between 1.0/1.1/1.3 at `temperature=0.7` changed the *flavor* of the degeneration (word loops vs. language-switching vs. unpunctuated run-on sentences) but never fixed it — a strong signal the problem wasn't repetition penalty at all. Swept `temperature` directly (0.1, 0.2, 0.3, 0.5, 0.7) with the same prompt: every value `>0` degenerated in some way; only `temperature=0.0` (greedy) was consistently coherent, matching the "gradual, not sudden" cosine-similarity noise (0.85–1.00, not 0.99–1.00) already documented in the FP16 recurrence fix above — small enough to never flip which token *wins* argmax, large enough to visibly distort the probability tail that stochastic sampling draws its variety from. Mitigated (not fixed) the same way as the `repetition_penalty` default: `VTEModel._default_temperature()` returns `0.0` for `qwen35`, `0.7` unchanged for Qwen2.5/Granite. This is a real, accepted trade-off, not a resolved bug — Qwen3.5 responses are now deterministic (same prompt always produces the same answer) until the underlying per-layer FP16 precision loss is reduced enough that sampling above `temperature=0` stops being visibly risky.

## Performance

~69 tok/s decode-only (vs. Ollama's 77 tok/s on the identical GGUF, ~89.6%) — see [Performance § Qwen 3.5](PERFORMANCE.md#qwen-35-2b-hybrid-gated-deltanet) for the three kernel-fusion/occupancy experiments that measured as null results, and why the shared GEMV/FFN infrastructure (not the new Gated DeltaNet kernels specifically) is the real remaining ceiling.
