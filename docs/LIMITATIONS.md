# Known limitations

- `generate_batch` requires every prompt in a batch to have identical token length (lockstep only); padding + attention masking for mixed-length batches is unimplemented.
- Batch sizes above 4 currently regress throughput (see [Performance](PERFORMANCE.md)) — not yet root-caused beyond the Infinity Cache hypothesis.
- Persistent activation buffers used by the HIP Graph executor are sized for `seq_len=1` (the dominant decode case); a multi-token prefill through that executor could in principle overrun those buffers. The current code avoids this by processing prefill token-by-token even in HIP Graph mode, reusing the single decode graph — but the buffers themselves aren't yet partitioned by shape class to make this impossible by construction rather than by calling convention.
- FFN kernel fusion and WMMA/Tensor Cores remain measured-and-rejected (former) or unevaluated in the current batch>1 regime (latter) — see [Performance](PERFORMANCE.md).
- `fused_gate_up_silu_kernel` (`VTE_ENABLE_FFN_FUSION`, off by default) is currently broken (produces NaN) for the project's actual FFN weight format — it assumes FP16 weights but `ffn_gate`/`ffn_up` are stored raw Q4_K. This predates and is unrelated to the batch=1 performance regression that already kept this fusion disabled. See [Architecture: Why QKV projection is fused, and why FFN fusion is off](ARCHITECTURE.md#why-qkv-projection-is-fused-and-why-ffn-fusion-is-off) for the full root-cause and why fixing it in-kernel was deliberately rejected rather than attempted.
- The batched (batch=4) LM Head GEMV achieves only ~7.5% of theoretical memory bandwidth, notably worse than the FFN's ~25%. A grid-dispatch-order fix (making batch_idx the fastest-varying block index) was implemented, validated bit-exact, measured (~3% gain, statistically flat end-to-end), and reverted — see [Bugs found during development](BUGS.md) for the full measurement. Remaining ~15x gap vs. the bandwidth floor is attributed to per-block dispatch/scheduling latency across 607744 blocks, not cache locality; closing it further needs WMMA/Tensor Cores or speculative decoding, both out of scope for a kernel-level tweak.
- Single-sequence decode throughput (~100 tok/s, stable across generation length) is still below the GPU's theoretical memory-bandwidth ceiling (~250 tok/s) and the decode graph's own measured GPU-only ceiling (~130 tok/s) — the CPU/dispatch overhead, attention occupancy, and artificial pacing costs that used to dominate this gap have been found and fixed (sampler, LM Head, Split-KV, keep-alive pulse), so what remains is genuinely about GEMV kernel efficiency, not overhead around it. Not yet closed.
- `LMHead`'s HIP-Graph capture path assumes the decode graph always runs at `seq_len=1` (true for the current lockstep decode/prefill scheme) — a future multi-token-per-step scheme would need the LM Head capture logic revisited alongside it.
- Real GPU temperature on Windows only works on AMD hardware, and only if either the PMLog (`ADL2_New_QueryPMLogData_Get`, RDNA/RDNA2/RDNA3) or the legacy Overdrive5 (`ADL_Overdrive5_Temperature_Get`, pre-RDNA) API responds — see [vte/bridge/adl_bridge.py](../vte/bridge/adl_bridge.py). On a non-AMD GPU, or if both ADL calls fail, `GPUMonitor` reports `None` and the UI shows "N/A" rather than fabricate a number. VRAM and the model's idle-unload countdown are WMI-sourced and always real, refreshed every ~0.35s.
- The dashboard shows two separate VRAM numbers, deliberately not one: the large value is what VTE itself allocated (weights+KV+arena, deterministic, always equals the Weights/KV/Arena breakdown below it); "Sistema (dedicada)" is the whole system's dedicated GPU memory from WMI (matches Task Manager's GPU tab, but includes the desktop compositor, browser, etc.). An earlier version conflated the two into one number, which made it jump in ways that had nothing to do with VTE's own usage.
- Split-KV attention (see [Bugs found during development](BUGS.md)) is the default for single-sequence decode (opt-out via `VTE_DISABLE_ATTN_SPLITKV=1`) after a validation pass covering short/long generations, chunk-boundary crossings, mid-generation cancellation, and scratchpad reuse immediately after a cancellation — all produced correct, coherent output with `pytest` green throughout. It only affects `generate()`; `generate_batch()` (batch>1) uses the unmodified `flash_attention_kernel` and is untouched by this change. One caveat worth knowing: greedy (`temperature=0`) output can occasionally diverge from the pre-Split-KV baseline partway through a long generation — not a bug, see the bug entry for why (reordered-but-equivalent floating point, not incorrect floating point), and irrelevant at the project's actual default (`temperature=0.7`).
- The inter-token keep-alive pulse now defaults to `0.0` (disabled) — see [Bugs found during development](BUGS.md) for the measurement showing this is currently both faster and more stable than any nonzero value tested on this hardware/driver. `VTE_KEEPALIVE_PULSE_MS` (milliseconds, e.g. `"2.0"`) restores the old behavior without a code change if a future driver update, different hardware, or another GPU-heavy application running alongside VTE ever reintroduces the DPM-downclocking stutter this was originally built to prevent.
- The desktop UI's language toggle (pt/en, see [Multi-architecture support](GRANITE.md) for the flag-icon fix) only translates UI chrome (labels, tooltips, status text) — it re-renders currently visible dynamic text (metrics, lifecycle status) immediately from the last known values, but log lines and chat messages are never translated, since they're either raw backend output or real user/model content, not UI copy.
