# Known limitations

What's genuinely unfinished or unresolved right now, grouped by area. Each item links to the document with the full story where one exists.

## Batching and prefill

- `generate_batch` requires every prompt in a batch to have identical token length (lockstep only); padding + attention masking for mixed-length batches is unimplemented.
- Batch sizes above 4 currently regress throughput (see [Performance](PERFORMANCE.md)): not yet root-caused beyond the Infinity Cache hypothesis.
- Persistent activation buffers used by the HIP Graph executor are sized for `seq_len=1` (the dominant decode case); a multi-token prefill through that executor could in principle overrun those buffers. The current code avoids this by processing prefill token-by-token even in HIP Graph mode, reusing the single decode graph, but the buffers themselves aren't yet partitioned by shape class to make this impossible by construction rather than by calling convention.
- `LMHead`'s HIP-Graph capture path assumes the decode graph always runs at `seq_len=1` (true for the current lockstep decode/prefill scheme): a future multi-token-per-step scheme would need the LM Head capture logic revisited alongside it.

## Kernel efficiency and rejected optimizations

- FFN kernel fusion and WMMA/Tensor Cores remain measured-and-rejected (former) or unevaluated in the current batch>1 regime (latter). See [Performance](PERFORMANCE.md).
- `fused_gate_up_silu_kernel` (`VTE_ENABLE_FFN_FUSION`, off by default) is currently broken (produces NaN) for the project's actual FFN weight format: it assumes FP16 weights but `ffn_gate`/`ffn_up` are stored raw Q4_K. This predates and is unrelated to the batch=1 performance regression that already kept this fusion disabled. See [Architecture: Why QKV projection is fused, and why FFN fusion is off](ARCHITECTURE.md#why-qkv-projection-is-fused-and-why-ffn-fusion-is-off) for the full root-cause and why fixing it in-kernel was deliberately rejected rather than attempted.
- The batched (batch=4) LM Head GEMV achieves only ~7.5% of theoretical memory bandwidth, notably worse than the FFN's ~25%. A grid-dispatch-order fix (making batch_idx the fastest-varying block index) was implemented, validated bit-exact, measured (~3% gain, statistically flat end-to-end), and reverted. See [Bugs found during development](BUGS.md) for the full measurement. Remaining ~15x gap vs. the bandwidth floor is attributed to per-block dispatch/scheduling latency across 607744 blocks, not cache locality; closing it further needs WMMA/Tensor Cores or speculative decoding, both out of scope for a kernel-level tweak.
- Single-sequence decode throughput (~100 tok/s, stable across generation length) is still below the GPU's theoretical memory-bandwidth ceiling (~250 tok/s) and the decode graph's own measured GPU-only ceiling (~130 tok/s): the CPU/dispatch overhead, attention occupancy, and artificial pacing costs that used to dominate this gap have been found and fixed (sampler, LM Head, Split-KV, keep-alive pulse), so what remains is genuinely about GEMV kernel efficiency, not overhead around it. Not yet closed.
- Split-KV attention (see [Bugs found during development](BUGS.md)) is the default for single-sequence decode (opt-out via `VTE_DISABLE_ATTN_SPLITKV=1`) after a validation pass covering short/long generations, chunk-boundary crossings, mid-generation cancellation, and scratchpad reuse immediately after a cancellation: all produced correct, coherent output with `pytest` green throughout. It only affects `generate()`; `generate_batch()` (batch>1) uses the unmodified `flash_attention_kernel` and is untouched by this change. One caveat worth knowing: greedy (`temperature=0`) output can occasionally diverge from the pre-Split-KV baseline partway through a long generation: not a bug, see the bug entry for why (reordered-but-equivalent floating point, not incorrect floating point), and irrelevant at the project's actual default (`temperature=0.7`).
- The inter-token keep-alive pulse now defaults to `0.0` (disabled). See [Bugs found during development](BUGS.md) for the measurement showing this is currently both faster and more stable than any nonzero value tested on this hardware/driver. `VTE_KEEPALIVE_PULSE_MS` (milliseconds, e.g. `"2.0"`) restores the old behavior without a code change if a future driver update, different hardware, or another GPU-heavy application running alongside VTE ever reintroduces the DPM-downclocking stutter this was originally built to prevent.

## Telemetry and desktop UI

- Real GPU temperature on Windows only works on AMD hardware, and only if either the PMLog (`ADL2_New_QueryPMLogData_Get`, RDNA/RDNA2/RDNA3) or the legacy Overdrive5 (`ADL_Overdrive5_Temperature_Get`, pre-RDNA) API responds. See [vte/bridge/adl_bridge.py](../vte/bridge/adl_bridge.py). On a non-AMD GPU, or if both ADL calls fail, `GPUMonitor` reports `None` and the UI shows "N/A" rather than fabricate a number. VRAM and the model's idle-unload countdown are WMI-sourced and always real, refreshed every ~0.35s.
- The dashboard shows two separate VRAM numbers, deliberately not one: the large value is what VTE itself allocated (weights+KV+arena, deterministic, always equals the Weights/KV/Arena breakdown below it); "Sistema (dedicada)" is the whole system's dedicated GPU memory from WMI (matches Task Manager's GPU tab, but includes the desktop compositor, browser, etc.). An earlier version conflated the two into one number, which made it jump in ways that had nothing to do with VTE's own usage.
- The desktop UI's language toggle (pt/en, see [Multi-architecture support](GRANITE.md) for the flag-icon fix) only translates UI chrome (labels, tooltips, status text): it re-renders currently visible dynamic text (metrics, lifecycle status) immediately from the last known values, but log lines and chat messages are never translated, since they're either raw backend output or real user/model content, not UI copy.

## Hardware portability

- **RDNA2 now has precompiled AOT kernels, but is still theoretical (architecture detection + CU-count scaling verified by simulation on the RX 7600, never run on real RDNA2 hardware).** GPU architecture targeting is dynamic: `get_gpu_architecture()` (`vte/bridge/hip_runtime.py`) reads the real device name at runtime and `GPU_ARCH_MAP` (`vte/config.py`) lists RX 6400 through RX 6950 XT (gfx1030/1031/1032/1034 — Navi21/22/23/24). CU-count scaling is split across two kernels with different status: **Split-KV attention** (`vte/core/split_kv_attention.py`) is genuinely dynamic: `_chunk_size_for_cus()` derives its grid from `HIPRuntime.get_num_cus()` (real `hipDeviceGetAttribute` value, WGP-doubled), not a fixed `32`. Simulated via `VTE_NUM_CUS=28` (the RX 6600, the one RDNA2 card in `GPU_ARCH_MAP` with fewer CUs than the RX 7600's 32) on the real RX 7600 hardware: chunk_size correctly shifted from 32 to 37, output stayed correct, and throughput was statistically flat (98.8 → 96.7 tok/s, ~2%). The scaling logic itself doesn't regress. Note this only validates the *scheduling logic*, not real RDNA2 throughput: a real RX 6600 also differs in memory bandwidth and clocks, which can't be simulated on RX 7600 silicon. **Split-K QKV** (`vte/core/fused_qkv_dispatch.py`) still hardcodes `chunk_size = hidden_size // 32` and a `grid=(32, ...)`: this is a fixed algorithmic split, not tied to the actual CU count for *correctness* (any CU count still produces the right numbers, just with imperfect occupancy on GPUs with other-than-32 CUs) — not migrated to `get_num_cus()` yet, deliberately left as a follow-up performance item rather than touched now, since changing a numerically-validated reduction kernel without RDNA2 hardware to re-verify against is a correctness risk this project isn't willing to take blind (and the RX 7600 baseline must not regress). Separately, an unrecognized GPU name (one not in `GPU_ARCH_MAP`, e.g. an RDNA3.5 APU) silently falls back to `DEFAULT_GPU_ARCH = "gfx1100"` (RDNA3), which would likely fail outright on real RDNA2 hardware since the two are different instruction sets.
- **No multi-GPU device selection: `HIPRuntime.initialize()` always calls `hipSetDevice(0)`, with no way to pin a specific device.** `vte/bridge/hip_runtime.py:333` hardcodes device index 0; there is no `ROCR_VISIBLE_DEVICES`/`HIP_VISIBLE_DEVICES` handling anywhere in the codebase, and no logic to prefer a discrete GPU over an integrated one. On a single-dGPU machine (the only configuration this project has ever run on: one RX 7600, nothing else) this is a non-issue — device 0 is the only device. On a machine with both a dGPU and an AMD iGPU, which HIP enumerates as device 0 is undefined from VTE's point of view; it could silently pick the iGPU. This surfaced as a real, severe bug in Lemonade itself (not in VTE — flagged here as a heads-up while reviewing a Lemonade GitHub issue during the VTE backend PR conversation): a dGPU+iGPU system's spawned `llama-server` child process never received the parent's `ROCR_VISIBLE_DEVICES` pin (the subprocess launcher only forwards a fixed allowlist of env vars per-backend, none of them GPU-selection-related for the ROCm path), so it enumerated both devices, the iGPU participated in device selection, and the mismatched kernel image crashed hard enough to hang the GPU. If VTE is ever run through Lemonade (or standalone) on a dGPU+iGPU AMD system, the same class of failure is plausible on the VTE side too: even if Lemonade correctly forwards `ROCR_VISIBLE_DEVICES` to `vte-server`'s subprocess, `HIPRuntime` doesn't read or honor it anywhere. Not fixed here: no dGPU+iGPU hardware exists in this project to design or validate a fix against, and guessing at device-selection logic without measuring it on real hardware would repeat exactly the mistake this project's own "measure, don't guess" discipline exists to avoid. Revisit when such hardware becomes available.
- **ROCm 6.4 → 7.1 upgrade validated on real hardware (RX 7600, gfx1102): no regression found.** Ran the full pytest suite (68 tests), a bit-exact regression across all 4 registered models, a 250-token generation (crosses Split-KV chunk boundaries), and a batch=4 `generate_batch` run, all against a real ROCm 7.1 install side-by-side with the 6.4 baseline. Results: device properties, `get_num_cus()` (still correctly WGP-doubled), and every precompiled `.hsaco`'s SHA256 hash were byte-identical between versions (the AOT cache loaded and ran without triggering the recompile fallback even once); the 3 pytest failures present are identical on both versions (pre-existing test-fixture bugs unrelated to ROCm, not a regression); generation output was bit-exact and tok/s statistically flat. Also reconfirmed the `hipGetDeviceProperties`/R0000 ABI behavior documented above is unchanged on 7.1 (byte-identical output calling the plain symbol vs. `hipGetDevicePropertiesR0000` explicitly). One real bug surfaced by the upgrade itself, not by ROCm: `vte/bridge/dll_discovery.py` hardcoded the DLL filename to `amdhip64.dll`/`amdhip64_6.dll`, so it silently would not have found `amdhip64_7.dll` (ROCm 7.1 ships no unversioned alias, confirmed 6.4 doesn't either — this naming scheme, `amdhip64_<major>.dll` with no stable alias, appears intentional and consistent across both installed versions, not an AMD defect). Fixed to glob for any `amdhip64_*.dll`, preferring the newest version. This is a single-datapoint result (one card, one version jump); it doesn't prove every future ROCm release will be this clean, only that this one was, measured rather than assumed.
- **Precompiled AOT kernels for `gfx1100`/`gfx1101` (RX 7900/7600 XT/7700/7800 series) and `gfx1030`/`1031`/`1032`/`1034` (RDNA2: RX 6900 XT down to RX 6400) compile cleanly but were never run.** `scripts/build_kernel_cache.py` warms the real kernel cache by actually loading and generating on every registered model on the real `gfx1102` hardware (RX 7600): those binaries are genuinely validated. `scripts/cross_compile_kernel_cache.py` (RDNA3 siblings) and `scripts/cross_compile_rdna2_kernel_cache.py` (RDNA2) both extend this by reusing the exact rendered `.hip` sources from that real run and invoking `hipcc --offload-arch=<target>` directly, without loading or executing the result (no such hardware exists to test on). All 63 kernels currently needed by the 4 registered models compiled without error for all 6 targets (`gfx1100`, `gfx1101`, `gfx1030`, `gfx1031`, `gfx1032`, `gfx1034`), but "compiles cleanly" only proves the source is syntactically valid C++ for that ISA, not that it produces correct numbers on it; a subtle codegen difference between architecture sub-variants (or generations) could in principle miscompile silently. `HIPRuntime.load_kernel_safe()`'s automatic fallback (see the bug entry above) only catches a binary that *fails to load* (ABI/driver mismatch): it cannot detect one that loads fine and computes something wrong. RDNA2 was initially excluded from cross-compilation over concern about real ISA divergence from RDNA3 — RDNA2 lacks RDNA3's AI-oriented ISA additions, specifically matrix/WMMA instructions and VOPD (dual-issue VALU) — before writing the RDNA2 script, this was checked against the actual code rather than assumed: an exhaustive grep of all 32 kernel templates (`vte/compiler/templates/*.hip.template`) found zero uses of matrix instructions or any low-level `__builtin_amdgcn_*` intrinsic; the only wave-level primitives used are `warpSize`/`__shfl_down`/`__shfl_xor`, which resolve identically on both generations since RDNA2 and RDNA3 are both wave32 (unlike GCN/CDNA's wave64). VOPD needs no equivalent guard: it isn't invoked via any HIP intrinsic, it's an instruction-scheduling decision the LLVM/AMDGPU backend makes on its own based on `--offload-arch`, and `compile_source_for_arch()` (`vte/compiler/codegen.py`) passes only `hipcc --genco --offload-arch=<target> -O3`, no manual `-mllvm`/`-mcpu`/feature flags that could force RDNA3-only codegen onto a gfx10 target; all 4 RDNA2 targets compiling without error is itself evidence the backend selected gfx10-appropriate (non-VOPD) encoding, since an attempt to emit VOPD for a target that doesn't support it would fail to assemble rather than silently miscompile. None of this eliminates the general risk (identical HIP source can still codegen subtly differently per architecture in ways unrelated to matrix instructions or VOPD), but it removes the two *known*, specific ISA divergences this project was worried about between these two generations.

## Reverted features and VRAM management

- **Speculative decoding was implemented and then reverted.** A draft-model-based approach (Qwen2.5 0.5B proposing tokens for the 1.5B/7B targets, verified in a batched forward pass) and a follow-up Prompt Lookup Decoding variant (n-gram search over the token history instead of a draft model) were both built, made numerically correct, and measured: neither beat the non-speculative baseline on this GPU (best case ~0.5x of baseline throughput even after fixing real bugs along the way: a global raw-weight-registry leak between coexisting models, a `hipDeviceReset()` destroying a second model's VRAM on unload, a missing kernel argument in the batched LM Head, and GPU clock/P-state dropping between bursty CPU/GPU handoffs). The real, independent bug fixes found during that work were kept (see [Bugs found during development](BUGS.md)); the speculative decoding code itself was removed rather than shipped as a slower default path.
- **VRAM budgeting is now cross-model-aware (the 95% check and cleanup used to be per-instance, not per-process. See [Bugs found during development](BUGS.md) for both bugs this caused), but each loaded `VTEModel` still gets its own `SlabAllocator` pool sized only for its own weights/KV-cache/arena: there's no shared pool, just a shared safety counter.** `HIPRuntime.safe_malloc()`'s "never exceed 95% VRAM" check (`VRAM_SAFETY_MARGIN=0.95` in `vte/config.py`) now sums bytes across every `HIPRuntime` instance in the process (`HIPRuntime._process_wide_allocated_bytes`, a class-level counter) and cross-checks against the GPU's real free memory (`hipMemGetInfo`, catches VRAM used by other processes too: games, browsers, etc.), so a second model loading on top of an already-resident one correctly counts against the same 95% ceiling instead of each instance seeing the full budget for itself. A `VRAMPressureGuard` (`vte/bridge/vram_pressure_guard.py`) also polls real GPU VRAM in the background and auto-unloads the least-recently-used loaded model if usage stays above 95%. `SlabAllocator` itself still never returns VRAM to the OS/driver once `hipMalloc`'d (freed blocks go back to an internal free-list for reuse within that same pool, not `hipFree`'d): unloading a model frees its own pool via `unload()`, and (since the `hipDeviceReset()` fix) no longer disturbs any other model's pool still resident in the same process.

## Model compatibility

- Only the Bartowski Q6_K conversion of Qwen 3.5 2B is validated to work: a structurally different Unsloth Q4_K_M conversion of the same nominal model produced immediate incoherence and is not registered. Full details in [Qwen 3.5: Known limitation: only the Bartowski Q6_K conversion is validated](QWEN35.md#known-limitation-only-the-bartowski-q6_k-conversion-is-validated-to-work).
